# 新人指南：5 分钟读懂这套方案

## 1. 这是什么

这套工程是一套面向多种业务 Agent 的通用安全底座。

它要同时回答三件事：

- 用户是谁
- 当前操作的 Agent 是谁
- 这个 Agent 现在能访问哪些资源

## 2. 先记住最重要的 4 句话

1. 用户入口是：`ALB -> 接入Agent`
2. `Agent网关` 是内部安全编排服务
3. 接入 Agent 不直接对接 `IDaaS`
4. 接入 Agent 后端/BFF 最终只拿：
   - `security_session_id`
   - `TR`

## 3. 先记住 7 个模块

- **ALB**
  - 浏览器入口流量层
- **接入 Agent**
  - 用户真正使用的业务 Agent
  - 前端负责页面和跳转
  - 后端/BFF负责调用 `Agent网关` 和 `MCP`
- **Agent网关**
  - 登录、授权、`Tc/T1/TR` 的统一编排层
- **IDaaS**
  - 用户登录和用户授权中心
- **IAM**
  - 签发 `T1` 和 `TR`
- **Agent策略中心**
  - 维护“策略 code <-> MCP 工具”映射
- **MCP网关 / MCP Server**
  - 真正的资源访问层

## 4. 三个 Token 是主线

- **Tc**
  - 用户令牌
  - 表达“用户是谁、用户授权了哪些策略 code”

- **T1**
  - Agent 令牌
  - 表达“Agent 是谁”

- **TR**
  - 资源令牌
  - 表达“这个 Agent 现在代表这个用户，可以访问哪些资源”

一句话记忆：

- `Tc` 是用户授权结果
- `T1` 是 Agent 身份结果
- `TR` 是最终资源访问结果

真正访问 MCP 时，只能用 `TR`。

## 5. 当前主流程怎么走

1. 用户先进入接入 Agent 页面
2. 用户第一次发起资源型请求时，接入 Agent 后端/BFF 调 `Agent网关`
3. `Agent网关` 判断当前是否需要：
   - 确认用户身份
   - 请求用户授权
   - 直接返回可用 `TR`
4. 如果需要前置动作，接入 Agent 前端只负责按网关给的 `redirect_url` 跳转
5. `Agent网关` 自己负责和 `IDaaS`、`IAM` 打交道
6. `Agent网关` 最终把：
   - `security_session_id`
   - `TR`
   返回给接入 Agent 后端/BFF
7. 接入 Agent 后端/BFF 再拿 `TR` 去访问 MCP

这里的 `security_session_id` 不是浏览器站点登录 session，而是 `Agent网关` 创建并维护的安全会话号。

## 6. Web Agent 为什么像有两层“登录/认证”

### 第一层：站点登录

解决：

`用户能不能进入 Web Agent 网站`

### 第二层：Agent 资源认证授权

解决：

`这个 Agent 能不能代表用户访问 MCP 资源`

所以更准确地说，不是“登录两次网站”，而是：

- 先登录网站
- 再确认 Agent 是否可以代表用户访问资源

如果站点登录和 `IDaaS` 共用企业 SSO，第二层很多时候不需要再次输入密码。

## 7. 为什么需要 Agent 网关

因为如果没有它，每个接入 Agent 都要自己处理：

- 登录启动
- 登录回调
- 授权启动
- 授权回调
- `Tc`
- `T1`
- `TR`

所以 `Agent网关` 的核心价值是：

`把复杂的安全流程从接入 Agent 里抽出来，做成统一安全中台。`

## 8. 为什么需要 Agent 策略中心

因为用户授权的是“策略 code”，而系统执行的是“MCP 工具”。

例如：

- 用户授权：`erp:report:read`
- 实际调用：`mcp:financial-report-server/query_monthly_report`

所以必须有一个中间层负责映射：

- `tool -> code`
- `code -> tool`

这就是 `Agent策略中心` 的作用。
