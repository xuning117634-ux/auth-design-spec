# 数据流转说明

这份文档描述的是当前标准方案的数据流转口径。

请先记住一个前提：

- 浏览器流量入口：`ALB -> 接入Agent`
- `Agent网关` 是内部安全编排服务
- 接入 Agent 不直接对接 `IDaaS`
- 登录、授权、`Tc / T1 / TR` 的申请、校验与刷新都由 `Agent网关` 负责
- 接入 Agent 前端只负责跳转和回到原页面
- 接入 Agent 后端/BFF负责调用 `Agent网关`、调用 `MCP`，并保存本地安全上下文映射

## 1. 先区分两层能力

Web Agent 场景下，很容易把两层流程混在一起：

### 第一层：站点登录

这层解决：

`用户能不能进入 Web Agent 网站`

产物通常是：

- 浏览器 cookie
- `site_session_id`
- 站点自己的登录态

### 第二层：Agent 资源认证授权

这层解决：

`这个 Agent 能不能代表当前用户访问 MCP 资源`

这层才会产生：

- `request_id`
- `Tc`
- `T1`
- `TR`
- `security_session_id`

后面本文件重点描述的是第二层。

## 2. 阶段 0：接入准备

在正式跑业务流之前，接入 Agent 需要先完成基础接入。

### 2.1 Agent 注册

接入 Agent 向 `Agent网关` 注册：

- `agent_id`
- `agent_name`
- `appId`
- `version`
- `subscribed_tools`
- 可选 `callback_base_url`

这一步的结果是：

- `Agent网关` 知道这是哪个 Agent
- `IAM` 后续可以基于这个 `agent_id` 签发 `T1`
- `Agent策略中心` 可以围绕这组订阅工具做策略映射

这里的 `callback_base_url` 不是运行时主流程必需字段，只用于：

- 回跳白名单校验
- 默认回跳地址兜底

运行时主链路依赖的是 `client_return_url`，它应该是**当前原聊天页的完整地址**。

## 3. 阶段 1：用户进入接入 Agent

这一阶段的流量链路是：

`用户浏览器 -> ALB -> 接入Agent前端 -> 接入Agent后端/BFF`

此时有两种情况：

- 如果 Web Agent 自己有站点登录体系，则先完成站点登录
- 如果当前请求还没有触发资源访问，则不一定立即进入 Agent 资源认证授权流程

也就是说：

`用户打开页面` 不等于 `立刻生成 TR`

通常是到用户第一次发起资源型请求时，接入 Agent 后端/BFF 再去调用 `Agent网关`。

## 4. 阶段 2：接入 Agent 发起 `TR` 获取流程

当用户在页面里发起真正需要访问 MCP 的请求时，接入 Agent 后端/BFF 调用：

`POST /api/v1/runtime/tr/acquire`

这一步的目标是判断：

- 当前用户是否需要先确认身份
- 当前用户是否需要先授权
- 当前会话是否已经具备可用的 `TR`

同时，`Agent网关` 会为这次调用创建一条内部 `security_request` 记录。后续：

- 登录回调
- 授权回调
- 再次获取 `TR`
- 安全会话创建

都围绕这条 `security_request` 推进。

### 4.1 默认方案下，权限点怎么来

默认方案是主流程。

`Agent网关` 会根据接入 Agent 注册时上报的 `subscribed_tools`，调用 `Agent策略中心` 做一次：

`tool -> code`

反查，得到当前 Agent 对应的整组候选策略 code。

因此默认方案的特点是：

- 适配最广
- 接入 Agent 改动最小
- 用户一次看到的是当前 Agent 订阅工具对应的整组授权项

这里有一个明确边界：

`Agent网关` 只有在确认当前 security_request 已绑定用户身份之后，才会去 Agent策略中心 查询这组候选权限点。

## 5. 阶段 3：身份确认与登录回跳

如果 `Agent网关` 判断当前缺少用户身份确认，会返回：

- `REDIRECT_REQUIRED`
- `request_id`
- 一个登录启动 `redirect_url`

这里要注意：

- 接入 Agent 不直接拼 `IDaaS` 地址
- 接入 Agent 前端只是按后端/BFF返回的 `redirect_url` 跳转
- 真正和 `IDaaS` 打交道的是 `Agent网关`
- 如果 `Agent网关` 已经能够复用第一层站点登录态，确认当前用户具备可复用的企业身份上下文，就不需要再发起登录跳转，而是直接进入授权阶段

### 5.1 推荐链路

1. `接入Agent后端/BFF -> Agent网关`
   发起 `TR` 获取请求
2. `Agent网关 -> 接入Agent后端/BFF`
   返回 `REDIRECT_REQUIRED + request_id + redirect_url`
3. `接入Agent后端/BFF -> 接入Agent前端`
   返回跳转指令
4. `接入Agent前端 -> 用户浏览器`
   发起重定向
5. `用户浏览器 -> Agent网关`
   打开登录启动地址，例如 `/api/v1/auth/login/start`
6. `Agent网关 -> 用户浏览器`
   302 到 `IDaaS`
7. `用户浏览器 -> IDaaS`
   完成登录
8. `IDaaS -> Agent网关`
   调用登录回调接口
9. `Agent网关 -> 用户浏览器`
   302 回到原聊天页 `client_return_url`
10. `用户浏览器 -> 接入Agent前端`
    回到业务页面
11. `接入Agent前端 -> 接入Agent后端/BFF`
    继续当前请求
12. `接入Agent后端/BFF -> Agent网关`
    再次调用 `POST /api/v1/runtime/tr/acquire`，并带上 `request_id`

这一阶段的结果是：

- `Agent网关` 知道当前用户是谁
- 当前安全请求进入下一状态
- 接入 Agent 后端/BFF 后续继续调用同一个 `TR` 获取接口，并带上 `request_id`

## 6. 阶段 4：用户授权与 `Tc`

如果用户还没有对当前 Agent 完成资源授权，`Agent网关` 会返回：

- `REDIRECT_REQUIRED`
- `request_id`
- 一个授权启动 `redirect_url`

### 6.1 授权链路

1. `接入Agent后端/BFF -> Agent网关`
   再次调用 `POST /api/v1/runtime/tr/acquire`，并带上 `request_id`
2. `Agent网关 -> 接入Agent后端/BFF`
   返回 `REDIRECT_REQUIRED + request_id + redirect_url`
3. `接入Agent前端 -> 用户浏览器`
   发起重定向
4. `用户浏览器 -> Agent网关`
   打开授权启动地址，例如 `/api/v1/auth/consent/start`
5. `Agent网关 -> 用户浏览器`
   302 到 `IDaaS` 授权页
6. `用户浏览器 -> IDaaS`
   用户确认授权
7. `IDaaS -> Agent网关`
   调用授权回调接口
8. `Agent网关 -> IDaaS`
   用授权码换取 `Tc`
9. `IDaaS -> Agent网关`
   返回 `Tc`
10. `Agent网关 -> 用户浏览器`
    302 回到原聊天页 `client_return_url`
11. `接入Agent前端 -> 接入Agent后端/BFF`
    继续当前请求
12. `接入Agent后端/BFF -> Agent网关`
    再次调用 `POST /api/v1/runtime/tr/acquire`，并带上 `request_id`

### 6.2 `Tc` 表达什么

`Tc` 只表达两件事：

- 用户是谁
- 用户最终授权了哪些策略 code

默认方案下，`Tc.consented_scopes` 往往是当前 Agent 整组候选权限点。

## 7. 阶段 5：`T1` 获取、`TR` 生成与安全会话创建

当 `Agent网关` 已经拿到 `Tc` 后，会继续走下面这条内部链路：

1. 向 `IAM` 调用真实接口：
   `POST https://apig.hisuat.huawei.com/iam/projects/{proxy_project_id}/assume_agent_token`
   获取 `T1`
2. 读取 `Tc.consented_scopes`
3. 调用 `Agent策略中心` 做：
   `code -> tool`
   正向查询
4. 将映射出的工具集合与 `agent_registry.subscribed_tools` 做交集
5. 向 `IAM` 调用真实接口：
   `POST https://apig.hisuat.huawei.com/iam/auth/resource-token`
   生成最终的 `TR`
6. 将 IAM 返回的 `refresh_token` 一并保存在网关侧安全会话中
7. 首次创建一条 `security_session`

### 7.1 三个 Token 在这一阶段的分工

- `Tc`
  - 用户授权输入
- `T1`
  - Agent 身份输入
- `TR`
  - 最终资源访问输出

### 7.2 `TR` 返回给谁

这里非常关键：

- `Tc / T1` 不下发给接入 Agent
- 接入 Agent 后端/BFF最终只拿到：
  - `security_session_id`
  - `TR`

当 `POST /api/v1/runtime/tr/acquire` 返回 `READY` 时，表示这次流程已经具备访问资源的最小安全上下文。

此时 `Agent网关` 会首次创建一条安全会话，并把：

- `request_id`
- `agent_id`
- `user_id`
- `tc_ref`
- `current_tr_ref`
- `current_refresh_token_ref`
- `effective_resources`

绑定到同一条安全会话记录上。

## 8. 阶段 6：首次访问 MCP

接入 Agent 后端/BFF 拿到 `TR` 后，才会真正调用：

`POST /api/v1/mcp/invoke`

MCP 侧的判断逻辑是：

1. `MCP网关` 通过内部运行时校验接口检查 `TR`
2. 校验：
   - 是否过期
   - 是否有效
   - 当前工具是否在 `granted_resources` 中
3. 校验通过后再转给 `MCP Server`
4. `MCP Server` 根据：
   - 用户信息
   - 策略范围
   - 资源范围
   做数据过滤、脱敏和审计

说明：

- 当前 IAM 文档未提供 `TR introspect` 官方接口
- 因此当前方案保留内部 `TR` 运行时校验接口

## 9. 阶段 7：后续会话复用与 `TR` 刷新

用户继续多轮对话时，接入 Agent 后端/BFF 优先复用当前：

- `security_session_id`
- `TR`

但复用的定位键不是 `conversation_id`，而是：

`user_id + agent_id + site_session_id`

也就是说，同一浏览器站点会话下、同一用户、同一 Agent 的多个聊天页面，可以复用同一条安全会话。

如果 `TR` 还有效，直接访问 MCP。

如果 `TR` 过期或缺失，接入 Agent 后端/BFF 调用：

`POST /api/v1/runtime/security-sessions/{security_session_id}/tr/refresh`

这里要注意：

- `security_session_id` 只是会话索引，不是独立凭证
- 接入 Agent 调 `Agent网关` 时，仍然需要运行时身份认证
- 这一层认证机制当前先抽象保留，不绑定具体技术实现

### 9.1 刷新后的外部结果

对接入 Agent 后端/BFF 来说，刷新接口只暴露两类外部结果：

- `READY`
  - 成功拿到新的 `TR`
- `REDIRECT_REQUIRED`
  - 需要重新进入某个前置流程

`LOGIN_REQUIRED / CONSENT_REQUIRED` 仍然只保留在网关内部状态模型中。

在真实 IAM 对接下，刷新主路径优先走：

`POST https://apig.hisuat.huawei.com/iam/auth/refresh-resource-token`

只有当 `refresh_token` 不可用、但 `Tc` 仍有效时，才回退到重新获取 `T1 + TR`。

这里的关键点是：

- 浏览器回跳不直接把 `TR` 带给前端
- 接入 Agent 后端/BFF 通过 `request_id` 再次调用 `TR` 获取接口拿到首次 `READY`
- 之后再通过 `security_session_id` 走刷新链路
- `refresh_token` 始终只保留在 `Agent网关` 侧，不下发给接入 Agent

## 10. 总结

这套方案最关键的边界是：

- 浏览器流量先到接入 Agent
- 接入 Agent 不直接对接 `IDaaS`
- `Agent网关` 统一负责安全编排服务

所以整条主线可以压缩成一句话：

`用户先进入接入 Agent，接入 Agent 在需要访问资源时调用 Agent网关，Agent网关负责登录、授权和 Tc/T1/TR 生成，接入 Agent 后端/BFF 最终只拿 TR 去访问 MCP。`
