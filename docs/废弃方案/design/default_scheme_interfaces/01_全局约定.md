# 01. 全局约定

## 1. 角色边界

- **接入Agent前端**：页面展示、跳转和回到原页面，不持有 `security_session_id / tr_token`
- **接入Agent后端/BFF**：调用 `Agent网关`、调用 `MCP`，维护站点会话到安全上下文的本地映射
- **Agent网关**：统一安全编排层，负责登录、授权、`Tc / T1 / TR` 申请与刷新
- **IDaaS**：负责登录、授权、签发 `Tc`
- **IAM**：负责签发 `T1`、生成 `TR`、刷新 `TR`
- **Agent策略中心**：负责 `tool -> code` 和 `code -> tool` 映射
- **MCP网关**：资源访问入口，校验 `TR`
- **MCP Server**：真正执行工具调用并返回业务数据

## 2. 内部状态模型

这套接口与交互设计统一围绕下面 8 个对象展开：

| 对象 | 作用 | 关键字段 | 创建时机 |
|---|---|---|---|
| `agent_registry` | 保存接入 Agent 注册信息 | `agent_id`、`agent_name`、`subscribed_tools`、可选 `callback_base_url` | `POST /api/v1/agents/register` |
| `security_requests` | 保存一次 `TR` 获取流程的全过程状态 | `request_id`、`agent_id`、`user_request`、`client_return_url`、`status`、`gateway_user_context`、`tc_ref`、`security_session_id`、`effective_resources` | `POST /api/v1/runtime/tr/acquire` |
| `security_sessions` | 保存可复用的安全会话与当前 `TR` | `security_session_id`、`agent_id`、`user_id`、`current_tr_ref`、`current_refresh_token_ref`、`effective_resources` | 首次进入 `READY` 后 |
| `agent_runtime_bindings` | 保存接入 Agent 后端/BFF本地映射 | `user_id`、`agent_id`、`site_session_id`、`security_session_id`、`tr_token`、`expires_at` | 首次拿到 `READY` 后 |
| `tokens` | 保存 `Tc / T1 / TR / refresh_token` 的响应与 payload | `token_id`、`expires_at`、`access_token`、`payload`、`refresh_token` | 相关 token 生成时 |
| `policy_mappings` | 保存策略 code 和 MCP 工具映射 | `policy_code`、`server_id`、`tool_name` | 初始化加载 |
| `users` | 保存用户目录 | `user_id`、`username`、`department` | 初始化加载 |
| `mcp_data` | 保存工具执行结果样例 | `tool_name`、`result` | 初始化加载 |

其中最关键的是 `security_requests`。登录回调、授权回调、再次获取 `TR` 和首次安全会话创建，全部围绕它推进。

## 3. `security_requests` 状态推进规则

`security_requests` 是当前接口设计里的唯一流程主记录。对接入 Agent 来说，外层只暴露两类结果：

- `REDIRECT_REQUIRED`
- `READY`

网关内部仍然维护以下 4 类流程状态：

| 状态 | 含义 | 典型进入方式 |
|---|---|---|
| `LOGIN_REQUIRED` | 当前请求还缺少网关侧用户身份上下文 | 首次创建安全请求且未完成身份确认 |
| `CONSENT_REQUIRED` | 当前请求已完成身份确认，但还缺少用户授权 | 登录回调完成后，查询状态时进入 |
| `READY` | 当前请求已具备 `security_session_id + TR` | 授权成功、`Tc / T1 / TR` 全部完成后 |
| `FAILED` | 当前流程失败 | 例如用户拒绝授权 |

### 3.1 推荐状态机

```text
创建 security_request
    -> LOGIN_REQUIRED
        -> 登录回调成功
        -> CONSENT_REQUIRED
            -> 授权回调成功 + 获取 Tc
            -> 查询状态时内部生成 T1 / TR
            -> READY
            -> 授权拒绝
            -> FAILED
```

### 3.2 关键规则

- `POST /api/v1/runtime/tr/acquire` 是接入 Agent 唯一的获取/续取 `TR` 入口
- 浏览器回跳只负责让前端回到原页面，不直接把 `TR` 带给前端
- `Agent网关` 在 `LOGIN_REQUIRED` 阶段不会提前查询策略中心
- `Agent网关` 只有在确认 `request_id` 已绑定用户身份后，才会去做 `tool -> code` 反查

## 4. `security_sessions` 规则

- 安全会话只在 `security_request` 首次进入 `READY` 时创建
- 安全会话创建后，只有接入 Agent 后端/BFF 持有：
  - `security_session_id`
  - 当前 `TR`
- `refresh_token` 由 `Agent网关` 保存，不下发给接入 Agent
- 后续刷新 `TR` 时复用已有 `security_session_id`，不再新建安全会话
- 同一浏览器站点会话下，同一用户、同一 Agent 默认复用同一条安全会话

## 5. 接口分层冻结

### 5.1 第一层：接入 Agent 可见接口

- `POST /api/v1/runtime/tr/acquire`
- `POST /api/v1/runtime/security-sessions/{security_session_id}/tr/refresh`
- `POST /api/v1/mcp/invoke`

### 5.2 第二层：Agent 网关内部编排接口

- `GET /api/v1/auth/login/start`
- `GET /api/v1/auth/login/callback`
- `GET /api/v1/auth/consent/start`
- `GET /api/v1/auth/consent/callback`
- `POST /internal/v1/policies/resolve-by-tools`
- `POST /oauth2/token`
- `POST https://apig.hisuat.huawei.com/iam/projects/{proxy_project_id}/assume_agent_token`
- `POST https://apig.hisuat.huawei.com/iam/auth/resource-token`
- `POST https://apig.hisuat.huawei.com/iam/auth/refresh-resource-token`

### 5.3 第三层：资源访问内部接口

- `POST /internal/v1/tokens/tr/introspect`
- `POST /internal/v1/tools/invoke`

## 6. IAM 真实接口映射

本地测试环境统一采用 IAM 提供的测试地址：

- `https://apig.hisuat.huawei.com`

当前已经明确的 3 个真实 IAM 接口如下：

| 用途 | 方法 | 真实路径 |
|---|---|---|
| 获取 `T1` | `POST` | `/iam/projects/{proxy_project_id}/assume_agent_token` |
| 获取 `TR` | `POST` | `/iam/auth/resource-token` |
| 刷新 `TR` | `POST` | `/iam/auth/refresh-resource-token` |

说明：

- `T1` 对应 IAM 文档里的 App Token
- `TR` 对应 IAM 文档里的 Resource Token
- 刷新链路优先走 `refresh-resource-token`
- 当前 IAM 文档没有提供 `TR introspect` 接口，因此保留 `/internal/v1/tokens/tr/introspect` 作为内部运行时校验接口

## 7. 通用接口风格

所有接口统一使用 HTTP + JSON，响应包络如下：

```json
{
  "code": "0",
  "message": "success",
  "data": {}
}
```

约定如下：

- `code = "0"` 表示接口调用成功
- 业务流程状态放在 `data.status`
- 业务失败时，`code` 使用固定错误码，`message` 给出可读描述

## 8. 公共头

| Header | 是否必选 | 说明 |
|---|---|---|
| `Content-Type: application/json` | 是 | JSON 接口统一要求 |
| `X-Trace-Id` | 是 | 全链路追踪 ID |
| `X-Request-Id` | 否 | 调用方自带的请求 ID；未传时由服务端生成 |
| `X-Agent-Id` | 按需 | 接入 Agent 标识，外部调用网关时推荐显式传入 |
| `Authorization` | 按需 | 接入 Agent 运行时身份认证，占位为 `Bearer <agent-runtime-auth>`，本轮不绑定具体技术实现 |

## 9. 时间与 Token 约定

- 时间统一使用 ISO 8601 格式
- 文档中的 token 只使用占位字符串，不展开完整 JWT
- Token payload 结构定义以 [04_令牌设计_策略中心版.md](/D:/IDEA_Project/init_env/auth-design-spec/docs/design/04_令牌设计_策略中心版.md) 为准

## 10. 对外状态与内部状态

### 10.1 对接入 Agent 暴露的运行时状态

| 状态值 | 说明 |
|---|---|
| `REDIRECT_REQUIRED` | 当前流程还缺前置动作，前端需要按 `redirect_url` 跳转 |
| `READY` | 已拿到 `security_session_id + TR`，可直接访问资源 |

### 10.2 网关内部流程状态

| 状态值 | 说明 |
|---|---|
| `LOGIN_REQUIRED` | 需要先确认用户身份 |
| `CONSENT_REQUIRED` | 需要用户授权 |
| `READY` | 流程已完成 |
| `FAILED` | 当前流程失败 |

## 11. 错误码

| 错误码 | 说明 |
|---|---|
| `CONSENT_DENIED` | 用户拒绝授权 |
| `SESSION_NOT_FOUND` | `security_session_id` 不存在或无效 |
| `TR_EXPIRED` | `TR` 已过期 |
| `TOOL_NOT_GRANTED` | 当前 `TR` 未授权该工具 |
| `TOKEN_INVALID` | token 签名或语义校验失败 |
| `INTERNAL_ERROR` | 服务内部错误 |

## 12. 关键安全约束

- `security_session_id` 只是会话索引，不是独立凭证
- 接入 Agent 调用 `Agent网关` 时，除了 `security_session_id` 之外，还需要 Agent 运行时身份认证
- `Tc / T1` 只留在 `Agent网关`，不下沉到接入 Agent
- `refresh_token` 只留在 `Agent网关` 侧安全会话记录中
- `TR` 是唯一可被 MCP 网关接受的资源访问令牌
- `granted_resources` 是网关或 MCP 侧的运行时计算结果，不是 `TR` 原始 payload 字段
