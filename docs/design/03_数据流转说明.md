# 数据流转说明

这份文档描述的是这套方案的数据流转口径。

请先记住一个前提：

- 浏览器流量入口：`ALB -> 接入Agent`
- `Agent网关` 是内部安全编排服务
- 接入 Agent 不直接对接 `IDaaS`
- 登录、授权、`Tc / T1 / TR` 的申请、校验与刷新都由 `Agent网关` 负责

## 1. 先区分两层能力

Web Agent 场景下，很容易把两层流程混在一起：

### 第一层：站点登录

这层解决：

`用户能不能进入 Web Agent 网站`

产物通常是：

- 站点自己的登录态
- Web session / cookie

### 第二层：Agent 资源认证授权

这层解决：

`这个 Agent 能不能代表当前用户访问 MCP 资源`

这层才会产生：

- `Tc`
- `T1`
- `TR`

后面本文件重点描述的是**第二层**。

## 2. 阶段 0：接入准备

在正式跑业务流之前，接入 Agent 需要先完成基础接入。

### 2.1 Agent 注册

接入 Agent 向 `Agent网关` 注册：

- `agent_id`
- `agent_name`
- `appId`
- `version`
- `subscribed_tools`
- `callback_base_url`

这一步的结果是：

- `Agent网关` 知道这是哪个 Agent
- `IAM` 后续可以基于这个 `agent_id` 签发 `T1`
- `Agent策略中心` 可以围绕这组订阅工具做策略映射

### 2.2 安全委托准备

接入 Agent 还需要和 `Agent网关` 建立基础信任关系，这样后续：

- 登录回调
- 授权回调
- `Tc / T1 / TR` 的申请、校验与刷新

都可以由 `Agent网关` 统一处理。

## 3. 阶段 1：用户进入接入 Agent

这一阶段的流量链路是：

`用户浏览器 -> ALB -> 接入Agent`

此时有两种情况：

- 如果 Web Agent 自己有站点登录体系，则先完成站点登录
- 如果当前请求还没有触发资源访问，则不一定立即进入 Agent 资源认证授权流程

也就是说：

`用户打开页面` 不等于 `立刻生成 TR`

通常是到用户第一次发起资源型请求时，接入 Agent 再去调用 `Agent网关`。

## 4. 阶段 2：接入 Agent 发起安全上下文检查

当用户在页面里发起真正需要访问 MCP 的请求时，接入 Agent 调用：

`POST /api/v1/runtime/tr/acquire`

这一步的目标不是拿数据，而是判断：

- 当前用户是否需要先确认身份
- 当前用户是否需要先授权
- 当前会话是否已经具备可用的 `TR`

同时，`Agent网关` 会为这次调用创建一条内部 `security_request` 记录，后续：

- 登录回调
- 授权回调
- 状态查询
- `session` 创建

都围绕这条 `security_request` 推进。

### 4.1 默认方案下，权限点怎么来

默认方案是主流程。

`Agent网关` 会根据接入 Agent 注册时上报的 `subscribed_tools`，调用 `Agent策略中心` 做一次：

`tool -> code`

反查，得到当前 Agent 对应的整组候选策略 code。

因此默认方案的特点是：

- 适配最广
- 接入 Agent 改动最小
- 用户一次看到的是当前 Agent 订阅工具对应的整组授权项

这里有一个明确边界：

`Agent网关` 只有在确认当前 `security_request` 已绑定用户身份之后，才会去 `Agent策略中心` 查询这组候选权限点。

## 5. 阶段 3：登录启动与登录回调

如果 `Agent网关` 判断当前缺少用户身份确认，会返回：

- `LOGIN_REQUIRED`
- 一个 `redirect_url`

这里要注意：

- 接入 Agent 不直接拼 `IDaaS` 地址
- 接入 Agent 只是把浏览器跳到 `Agent网关` 给出的 `redirect_url`
- 真正和 `IDaaS` 打交道的是 `Agent网关`

### 5.1 推荐链路

1. `接入Agent -> Agent网关`
   发起安全请求
2. `Agent网关 -> 接入Agent`
   返回 `REDIRECT_REQUIRED + request_id + redirect_url`
3. `接入Agent -> 用户浏览器`
   发起重定向
4. `用户浏览器 -> Agent网关`
   打开登录启动地址，例如 `/api/v1/auth/login/start`
5. `Agent网关 -> 用户浏览器`
   302 到 `IDaaS`
6. `用户浏览器 -> IDaaS`
   完成登录
7. `IDaaS -> Agent网关`
   调用登录回调接口
8. `Agent网关 -> 用户浏览器`
   302 回到接入 Agent 的 `client_return_url`
9. `用户浏览器 -> 接入Agent`
   回到业务页面
10. `接入Agent -> Agent网关`
    再次调用 `POST /api/v1/runtime/tr/acquire`，并带上 `request_id`

这一阶段的结果是：

- `Agent网关` 知道当前用户是谁
- 当前安全请求进入下一状态
- 接入 Agent 后续继续调用同一个 `TR` 获取接口，并带上 `request_id`

## 6. 阶段 4：用户授权与 `Tc`

如果用户还没有对当前 Agent 完成资源授权，`Agent网关` 会返回：

- `REDIRECT_REQUIRED`
- 一个授权启动 `redirect_url`

### 6.1 授权链路

1. `接入Agent -> Agent网关`
   再次调用 `POST /api/v1/runtime/tr/acquire`，并带上 `request_id`
2. `Agent网关 -> 接入Agent`
   返回 `REDIRECT_REQUIRED + request_id + redirect_url`
3. `接入Agent -> 用户浏览器`
   发起重定向
4. `用户浏览器 -> Agent网关`
   打开授权启动地址，例如 `/api/v1/auth/consent/start`
5. `Agent网关 -> 用户浏览器`
   302 到 `IDaaS` 授权页
6. `用户浏览器 -> IDaaS`
   用户确认授权
7. `IDaaS -> Agent网关`
   调用授权回调接口
8. `Agent网关 -> IDaaS`
   用授权码换取 `Tc`
9. `IDaaS -> Agent网关`
   返回 `Tc`
10. `Agent网关 -> 用户浏览器`
    302 回到接入 Agent 的 `client_return_url`
11. `接入Agent -> Agent网关`
    再次调用 `POST /api/v1/runtime/tr/acquire`，并带上 `request_id`

### 6.2 `Tc` 表达什么

`Tc` 只表达两件事：

- 用户是谁
- 用户最终授权了哪些策略 code

默认方案下，`Tc.consented_scopes` 往往是当前 Agent 整组候选权限点。

插件方案下，`Tc.consented_scopes` 会更窄，但当前文档不展开。

## 7. 阶段 5：`T1` 获取与 `TR` 生成

当 `Agent网关` 已经拿到 `Tc` 后，会继续走下面这条内部链路：

1. 向 `IAM` 调用本地测试环境真实接口：
   `POST https://apig.hisuat.huawei.com/iam/projects/{proxy_project_id}/assume_agent_token`
   获取 `T1`
2. 读取 `Tc.consented_scopes`
3. 调用 `Agent策略中心` 做：
   `code -> tool`
   正向查询
4. 将映射出的工具集合与 `agent_registry.subscribed_tools` 做交集
5. 向 `IAM` 调用本地测试环境真实接口：
   `POST https://apig.hisuat.huawei.com/iam/auth/resource-token`
   生成最终的 `TR`
6. 将 IAM 返回的 `refresh_token` 一并保存在网关侧会话中

### 7.1 三个 Token 在这一阶段的分工

- `Tc`
  - 用户授权输入
- `T1`
  - Agent 身份与工具订阅输入
- `TR`
  - 最终资源访问输出

### 7.2 `TR` 返回给谁

这里非常关键：

- `Tc/T1` 不下发给接入 Agent
- 接入 Agent 最终只拿到：
  - `session_id`
  - `TR`

当 `POST /api/v1/runtime/tr/acquire` 返回 `READY` 时，表示这次会话已经具备访问资源的最小安全上下文。

此时 `Agent网关` 会首次创建 `session`，并把：

- `request_id`
- `agent_id`
- `user_id`
- `tc_ref`
- `current_tr_ref`

绑定到同一条会话记录上。

同时，网关侧还会保存：

- `current_refresh_token_ref`

## 8. 阶段 6：首次访问 MCP

接入 Agent 拿到 `TR` 后，才会真正调用：

`POST /api/v1/mcp/invoke`

MCP 侧的判断逻辑是：

1. `MCP网关` 通过内部运行时校验接口检查 `TR`
2. 校验：
   - 是否过期
   - 是否有效
   - 当前工具是否在 `granted_resources` 中
3. 校验通过后再转给 `MCP Server`
4. `MCP Server` 根据：
   - 用户信息
   - 策略范围
   - 资源范围
   做数据过滤、脱敏和审计

说明：

- 当前 `docs/iam_api/api.md` 中未提供 `TR introspect` 官方接口
- 因此当前方案保留内部 `TR` 运行时校验接口

## 9. 阶段 7：后续会话复用与 `TR` 刷新

用户继续多轮对话时，接入 Agent 可以复用当前：

- `session_id`
- `TR`

如果 `TR` 还有效，直接访问 MCP。

如果 `TR` 过期或缺失，接入 Agent 调用：

`POST /api/v1/runtime/sessions/{session_id}/tr/refresh`

这里要注意：

- `session_id` 只是会话索引，不是独立凭证
- 接入 Agent 调 `Agent网关` 时，仍然需要运行时身份认证
- 这一层认证机制当前先抽象保留，不绑定具体技术实现

刷新时的三种典型结果是：

- `READY`
  - 成功拿到新的 `TR`
- `CONSENT_REQUIRED`
  - `Tc` 已失效，需要重新授权
- `LOGIN_REQUIRED`
  - 登录态或会话已无效，需要重新登录

在真实 IAM 对接下，刷新主路径优先走：

`POST https://apig.hisuat.huawei.com/iam/auth/refresh-resource-token`

也就是说，Agent 网关会读取当前 session 里保存的 `refresh_token` 去向 IAM 刷新 `TR`。只有当 `refresh_token` 不可用、但 `Tc` 仍有效时，才回退到重新获取 `T1 + TR`。

这里的关键点是：

- 浏览器回跳不直接把 `TR` 带给接入 Agent
- 接入 Agent 通过 `request_id` 再次调用 `TR` 获取接口拿到首次 `READY`
- 之后再通过 `session_id` 走刷新链路
- `refresh_token` 始终只保留在 Agent 网关侧，不下发给接入 Agent

## 10. 总结

这套方案最关键的边界是：

- 浏览器流量先到接入 Agent
- 接入 Agent 不直接对接 `IDaaS`
- `Agent网关` 统一负责安全编排服务

所以整条主线可以压缩成一句话：

`用户先进入接入 Agent，接入 Agent 在需要访问资源时调用 Agent网关，Agent网关负责登录、授权和 Tc/T1/TR 生成，接入 Agent 最终只拿 TR 去访问 MCP。`
