# 接口设计

`01` 为参考方案，`02/03/04/05` 为当前正式方案。

本文负责接口总览、统一约定和阅读顺序。每个模块的完整请求/响应示例放在 `04_接口设计明细` 目录中。

## 1. 适用范围

当前正式方案涉及 5 类接口：

1. 业务 Agent 对外接口。
2. Agent 网关接口。
3. 策略中心接口。
4. MCP 网关接口。
5. 外部依赖接口：
   - IDaaS
   - IAM
   - MCP 服务

## 2. 统一约定

### 2.1 鉴权材料

| 调用方向 | 鉴权材料 | 说明 |
| --- | --- | --- |
| 浏览器 -> 业务 Agent | `site_session_id` Cookie | 业务 Agent 自己的网站会话 |
| 浏览器 -> Agent 网关 | `request_id`、`gw_state`、一次性票据 | 浏览器只承载跳转恢复材料，不承载敏感令牌 |
| 业务 Agent -> Agent 网关 | 服务间鉴权 + 一次性票据 | 业务 Agent 后端用 `ticketST` 或 `token_result_ticket` 换取结果 |
| 业务 Agent -> MCP 网关 | `Authorization: Bearer <TR>` | 用户授权给 Agent 的权限点上限边界 |
| Agent 网关 -> 策略中心 | 服务间内网调用 | 由网络信任域或服务鉴权保护 |
| MCP 网关 -> 策略中心 | 服务间内网调用 | 由网络信任域或服务鉴权保护 |

### 2.2 关键对象

| 对象 | 作用 |
| --- | --- |
| `site_session_id` | 定位业务 Agent 本地 `site_session` |
| `ticketST` | base 登录完成后，业务 Agent 后端换取用户信息的一次性票据 |
| `token_result_ticket` | 业务授权完成后，业务 Agent 后端换取 `TR` 的一次性票据 |
| `request_id` | 一次“申请 TR / 补充授权”流程的请求编号 |
| `gw_state` | Agent 网关内部 OAuth state，不对业务 Agent 暴露 |
| `consented_scopes` | `Tc` 和 `TR.agency_user` 中携带的权限点对象数组，元素包含 `code + displayNameZh` |

### 2.3 浏览器 URL 安全约定

浏览器回跳业务 Agent 时只允许携带：

- `ticketST`
- `token_result_ticket`
- `request_id`
- `state`

浏览器 URL 中不得携带：

- `Tc`
- `TR`
- `access_token`
- `refresh_token`
- 用户工号、用户名、用户 UUID 等用户信息

### 2.4 统一错误响应建议

各模块接口建议统一返回：

```json
{
  "code": "REQUEST_FAILED",
  "message": "当前用户无权使用该 Agent 的此项功能：ERP 合同的可读权限",
  "traceId": "trc_20260417_001"
}
```

## 5. 当前正式结论

- 业务 Agent 只理解 `required_tools`、`site_session`、`TR` 和一次性票据交换。
- Agent 网关负责 `required_tools -> requiredPermissionPointCodes`。
- Agent 网关不维护长期用户登录态，不判断已有 `TR` 是否覆盖当前权限点。
- `Tc` 和 `TR.agency_user` 都携带 `consented_scopes`。
- `TR` 是用户授权给 Agent 的权限点上限边界。
- MCP 网关运行时先校验当前工具所需权限点是否在 `TR` 中，再做 Agent 策略判断，最后校验当前工具是否属于 `TR` 对应工具集合。
- 策略中心统一维护权限点定义、权限点与工具绑定关系、Agent 订阅权限点映射、Agent 策略。
- 权限点由 `permissionPointCode` 全系统唯一标识，上报时必须携带 `enterprise` 和 `appId`。
- 权限点查询必须携带 `enterprise`，`appId` 可选；不传 `appId` 表示查询该企业下全部应用的权限点。
