# 接口设计

`01` 为参考方案，`02/03/04/05` 为当前正式方案。

本文不再试图把所有接口细节压在一个文件里，而是采用：

- 本文负责：接口总览、统一约定、阅读顺序
- 明细文档负责：各模块接口、请求/响应示例、错误码与处理规则

## 1. 适用范围

当前正式方案涉及 5 类接口：

1. 业务 Agent 对外接口
2. Agent 网关接口
3. 策略中心接口
4. MCP 网关接口
5. 外部依赖接口
   - IDaaS
   - IAM
   - MCP 服务

## 2. 统一约定

### 2.1 鉴权材料

| 调用方向 | 鉴权材料 | 说明 |
| --- | --- | --- |
| 浏览器 -> 业务 Agent | `site_session_id` Cookie | 业务 Agent 自己的网站会话 |
| 浏览器 -> Agent 网关 | `gw_session_id` Cookie | 网关自己的登录会话 |
| 业务 Agent -> Agent 网关 | `Authorization: Bearer <gw_session_token>` | 网关与业务 Agent 之间的登录用户桥接凭证 |
| 业务 Agent -> MCP 网关 | `Authorization: Bearer <TR>` | 用户授权给 Agent 的权限点上限边界 |
| Agent 网关 -> 策略中心 | 服务间内网调用 | 由网络信任域或服务鉴权保护 |
| MCP 网关 -> 策略中心 | 服务间内网调用 | 由网络信任域或服务鉴权保护 |

### 2.2 关键对象

| 对象 | 作用 |
| --- | --- |
| `site_session_id` | 定位业务 Agent 本地 `site_session` |
| `gw_session_id` | 定位网关本地 `gw_session` |
| `gw_session_token` | 把业务 Agent 本地会话和网关已登录用户桥接起来 |
| `request_id` | 一次“申请 TR / 补充授权”流程的请求编号 |
| `authorizedPermissionPoints` | `Tc/TR` 中携带的权限点对象数组，元素包含 `code + displayNameZh` |

### 2.3 统一错误响应建议

各模块接口建议统一返回：

```json
{
  "code": "REQUEST_FAILED",
  "message": "当前用户无权使用该 Agent 的此项功能：ERP 合同的可读权限",
  "traceId": "trc_20260417_001"
}
```

其中：

- `code`：稳定错误码
- `message`：面向调用方的可读说明
- `traceId`：可选，便于排障

## 5. 当前正式结论

- 业务 Agent 只理解 `required_tools`、`gw_session_token`、`TR`
- Agent 网关负责 `required_tools -> requiredPermissionPointCodes`
- `Tc` 和 `TR` 都携带 `authorizedPermissionPoints`
- `TR` 是用户授权给 Agent 的权限点上限边界
- MCP 网关运行时先做 `TR` 范围校验，再做 Agent 策略判断
- 策略中心统一维护权限点定义、权限点与工具绑定关系、Agent 策略
