# 引入 Agent 网关 + 策略中心后的接口设计

`01` 为参考方案，`02/03/04/05` 为当前正式方案。

## 1. 目标

当前正式方案在 `Tc / T1 / TR` 三令牌模型基础上，进一步引入：

- `Agent 网关`
- `策略中心`

目标不是改变三令牌模型，而是重新划分模块职责：

- 浏览器 OAuth2 跳转和 callback 全部由 `Agent 网关` 处理
- `Tc / T1 / TR` 的申请、缓存、续期编排都由 `Agent 网关` 处理
- `策略中心` 统一维护：
  - 权限点定义
  - 权限点与工具绑定关系
  - Agent 策略
- 业务 Agent 不再上传 `scope/code`
- 业务 Agent 只上传本次请求需要的 `required_tools`
- 网关负责把 `required_tools` 反查成 `requiredPermissionPointCodes`
- 业务 Agent 带 `TR` 访问 `MCP 网关`
- `MCP 网关` 运行时先做 `TR.authorizedPermissionPoints` 范围校验，再做 Agent 策略判断

一句话总结：**业务 Agent 面向工具，Agent 网关面向授权流程，策略中心面向权限点和策略模型。**

## 2. 模块边界与开发职责

| 模块 | 主要职责 | 不负责什么 |
| --- | --- | --- |
| 业务 Agent 前端 | 页面入口、跳转处理、`sessionStorage` 恢复消息 | 不直连 `IDaaS / IAM / 策略中心`；不处理 code；不理解权限点内部结构 |
| 业务 Agent 后端/BFF | 建立站点登录态；判断请求要用哪些 `MCP tool`；按需向网关要 `TR`；带 `TR` 调 `MCP 网关` | 不实现 OAuth2 callback；不构造授权 URL；不申请 `Tc / T1 / TR` |
| Agent 网关 | 统一编排登录、授权、code 换 token、T1/TR 申请、状态维护；把 `required_tools` 翻译成 `requiredPermissionPointCodes` | 不承接业务聊天页流量；不直接代理 `MCP` 调用流量；不做运行时策略判断 |
| 策略中心 | 维护权限点定义、权限点与工具绑定关系、Agent 策略；提供查询接口 | 不处理浏览器跳转；不保存用户登录态；不申请令牌 |
| MCP 网关 | 接受 `TR`；校验当前工具依赖的权限点是否在 `TR.authorizedPermissionPoints` 内；查询 Agent 策略并做运行时判断；路由到 `MCP 服务` | 不处理登录授权跳转；不签发 `Tc / T1 / TR` |

## 3. 核心接口总览

| 调用方 | 被调用方 | 方法 | 路径 | 用途 | 关键输入 | 关键输出 |
| --- | --- | --- | --- | --- | --- | --- |
| 业务 Agent | Agent 网关 | `POST` | `/gw/token/resource-token` | 按需获取 `TR` 或跳转指令 | `gw_session_token`、`agent_id`、`required_tools`、`return_url`、`state` | `TR` 或 `redirect_url + request_id` |
| Agent 网关 | 策略中心 | `POST` | `/internal/v1/permission-points/resolve-by-tools` | 把工具集合翻译成本次所需权限点 code 集合 | `agent_id`、`required_tools` | `requiredPermissionPointCodes`、`permissionPoints` |
| MCP 网关 | 策略中心 | `POST` | `/internal/v1/permission-points/resolve-by-codes` | 根据权限点 code 反查候选工具 | `permissionPointCodes` | `tools` |
| MCP 网关 | 策略中心 | `POST` | `/internal/v1/agent-strategies/query` | 查询当前 Agent 对这些权限点的策略 | `agent_id`、`permissionPointCodes` | `strategies` |
| 用户浏览器 | Agent 网关 | `GET` | `/gw/auth/login` | 发起 base 登录 | `agent_id`、`return_url`、`state` | 302 到 IDaaS 登录页 |
| IDaaS | Agent 网关 | `GET` | `/gw/auth/base/callback` | base 登录成功回调 | `code`、`state` | 302 回业务 Agent 页 |
| 用户浏览器 | Agent 网关 | `GET` | `/gw/auth/authorize` | 发起业务授权 | `request_id` | 302 到 IDaaS 授权页 |
| IDaaS | Agent 网关 | `GET` | `/gw/auth/consent/callback` | 业务授权成功回调 | `code`、`state` | 302 回业务 Agent 页 |

## 4. 关键接口详细说明

### 4.1 `POST /gw/token/resource-token`

**调用方**：业务 Agent 后端/BFF  
**被调用方**：Agent 网关  
**用途**：当本地 `TR` 缺失、过期或无法覆盖本次所需工具集合时，向网关申请资源令牌；如果网关判断需要用户参与，则返回跳转指令

#### 请求示例

```http
POST /gw/token/resource-token
Authorization: Bearer gwst_001
Content-Type: application/json
```

```json
{
  "agent_id": "agt_business_001",
  "required_tools": [
    "mcp:financial-report-server/query_monthly_report",
    "mcp:financial-report-server/list_report_categories"
  ],
  "return_url": "https://business-agent.huawei.com/chat",
  "state": "st_auth_001"
}
```

#### 成功响应示例：返回 `TR`

```json
{
  "access_token": "eyJ...<TR-Payload>...signature",
  "expires_in": 3600
}
```

#### 成功响应示例：需要用户参与授权

```json
{
  "status": "redirect",
  "redirect_url": "https://agent-gateway.huawei.com/gw/auth/authorize?request_id=req_sec_001"
}
```

#### 处理规则

- 业务 Agent 上传的是 `required_tools`，不是 `scope/code`
- 网关先根据 `gw_session_token` 找到对应的 `gw_session_id`
- 再校验 `agent_id` 是否存在、状态是否可用，以及 `return_url` 是否满足回跳白名单约束
- 然后调用策略中心把 `required_tools` 反查成 `requiredPermissionPointCodes`
- 如果策略中心无法把这些工具解析成合法权限点，请求直接失败
- 如果当前 `gw_auth_context` 已覆盖这些权限点且已有有效 `TR`，直接返回 `TR`
- 如果没有，则返回 `redirect_url + request_id`

### 4.2 `POST /internal/v1/permission-points/resolve-by-tools`

**调用方**：Agent 网关  
**被调用方**：策略中心  
**用途**：根据本次业务请求需要访问的工具集合，反查需要向用户申请的权限点

#### 请求示例

```json
{
  "agentId": "agt_business_001",
  "requiredTools": [
    "mcp:financial-report-server/query_monthly_report",
    "mcp:financial-report-server/list_report_categories"
  ]
}
```

#### 响应示例

```json
{
  "requiredPermissionPointCodes": [
    "erp:report:r",
    "erp:report:category:list"
  ],
  "permissionPoints": [
    {
      "code": "erp:report:r",
      "displayNameZh": "ERP 报表的可读权限"
    },
    {
      "code": "erp:report:category:list",
      "displayNameZh": "ERP 报表分类的可读权限"
    }
  ]
}
```

### 4.3 `POST /internal/v1/permission-points/resolve-by-codes`

**调用方**：MCP 网关  
**被调用方**：策略中心  
**用途**：根据权限点 code 反查候选工具集合

### 4.4 `POST /internal/v1/agent-strategies/query`

**调用方**：MCP 网关  
**被调用方**：策略中心  
**用途**：查询当前 Agent 对这些权限点的策略

#### 请求示例

```json
{
  "agentId": "agt_business_001",
  "permissionPointCodes": [
    "erp:report:r",
    "erp:report:category:list"
  ]
}
```

## 5. 运行时行为

### 5.1 Agent 网关侧

Agent 网关负责：

1. 接收 `required_tools`
2. 反查 `requiredPermissionPointCodes`
3. 发起用户授权
4. 获取 `Tc / T1 / TR`
5. 把 `TR` 返回给业务 Agent

### 5.2 MCP 网关侧

MCP 网关负责：

1. 从 `TR.agency_user.authorizedPermissionPoints` 读取权限点对象
2. 提取当前请求工具所需的权限点 code
3. 校验这些 code 是否已包含在 `TR` 中
4. 查询策略中心拿到对应 Agent 策略
5. 执行运行时判断
6. 如果通过，再路由到目标 `MCP 服务`

这部分判定结论固定为：

- 即使策略放行，也必须先通过 `TR` 范围校验
- 只要当前工具依赖的权限点未在 `TR` 中，当前请求直接失败
- 只要当前工具依赖的权限点被 Agent 策略拒绝，当前请求直接失败

## 6. 本地缓存模型

业务 Agent 本地缓存统一包含：

- `current_tr`
- `covered_tools`
- `coveredPermissionPointCodes`
- `expires_at`

判断规则：

1. 新请求进入时，比较 `required_tools` 是否被 `covered_tools` 覆盖
2. 覆盖且未过期，直接复用 `current_tr`
3. 不覆盖或已过期，再向网关重新申请

## 7. 当前设计结论

- 业务 Agent 只理解工具、`TR`、`redirect_url`
- Agent 网关负责 `required_tools -> requiredPermissionPointCodes`
- `Tc` 和 `TR` 都携带 `authorizedPermissionPoints`
- `TR` 是用户授权给 Agent 的上限边界
- MCP 网关必须先做 `TR` 范围校验，再做 Agent 策略判断
