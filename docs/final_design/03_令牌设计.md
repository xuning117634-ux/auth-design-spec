# Agent 安全令牌设计（OAuth2/JWT）

`01` 为参考方案，`02/03/04/05` 为当前正式方案。

## 1. 说明

本文描述当前版本的三类令牌：

- `Tc`：用户令牌
- `T1`：Agent 令牌
- `TR`：资源令牌

当前版本中：

- `Tc` 和 `TR` 都携带 `authorizedPermissionPoints`
- 每个权限点对象至少包含：
  - `code`
  - `displayNameZh`
- `TR` 不是最终可调用工具列表
- `TR` 表达的是**用户授权给 Agent 的权限点上限边界**

## 2. 公共约定

### 2.1 标准 Claim

| Claim | 含义 |
|---|---|
| `iss` | 签发方 |
| `sub` | 令牌代表的主体 |
| `aud` | 目标接收方 |
| `nbf` | 生效时间 |
| `exp` | 过期时间 |
| `iat` | 签发时间 |
| `jti` | JWT 全局唯一 ID |

### 2.2 权限点对象

当前正式方案里，令牌中的权限点统一采用对象数组：

```json
"authorizedPermissionPoints": [
  {
    "code": "erp:report:r",
    "displayNameZh": "ERP 报表的可读权限"
  }
]
```

说明：

- `authorizedPermissionPoints` 是正式字段
- 运行时如果只需要 code，可从 `authorizedPermissionPoints[].code` 提取
- `coveredPermissionPointCodes` 这类纯 code 集合只作为缓存或判定过程中的派生集合出现

这样设计的收益：

- token 自描述更强
- 审计和展示更方便
- 前端或日志不需要额外反查中文描述
- 后续如要增加 `description`、`resource`、`action` 字段也容易扩展

## 3. Tc：用户令牌

`Tc` 由 `IDaaS` 签发，用来表达：

- 当前用户是谁
- 当前用户授权给 Agent 哪些权限点

### 3.1 接口说明

#### URI

```http
POST /oauth2/token
```

#### 请求体示例

```json
{
  "grant_type": "authorization_code",
  "code": "auth_code_001",
  "client_id": "agent_gateway_client",
  "redirect_uri": "https://agent-gateway.huawei.com/gw/auth/consent/callback"
}
```

#### 成功响应体示例

```json
{
  "code": "201",
  "message": "OK",
  "access_token": "eyJ...<Tc-Payload>...signature",
  "expires_at": "2026-04-08T11:00:00+08:00",
  "expires_in": 3600,
  "token_type": "Bearer"
}
```

### 3.2 Tc Payload 示例

```json
{
  "iss": "idaas",
  "sub": "z01062668",
  "aud": "agent-gateway",
  "nbf": "2026-04-08T10:00:00+08:00",
  "exp": "2026-04-08T11:00:00+08:00",
  "iat": "2026-04-08T10:00:00+08:00",
  "jti": "tc_001",
  "user": {
    "user_id": "z01062668",
    "uuid": "uid_001",
    "username": "张建国"
  },
  "authorizedPermissionPoints": [
    {
      "code": "erp:report:r",
      "displayNameZh": "ERP 报表的可读权限"
    },
    {
      "code": "erp:contract:r",
      "displayNameZh": "ERP 合同的可读权限"
    }
  ]
}
```

### 3.3 Tc 关键字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `user.user_id` | string | 用户工号 |
| `user.uuid` | string | 用户全局唯一标识 |
| `user.username` | string | 用户姓名 |
| `authorizedPermissionPoints` | object[] | 用户已授权给 Agent 的权限点对象数组 |
| `authorizedPermissionPoints[].code` | string | 权限点 code |
| `authorizedPermissionPoints[].displayNameZh` | string | 权限点中文描述 |

## 4. T1：Agent 令牌

`T1` 由 `IAM` 签发，用来表达：

- 当前 Agent 是谁
- IAM 是否认可该 Agent 以程序集成身份发起资源访问

### 4.1 T1 Payload 示例

```json
{
  "iss": "iam",
  "sub": "agt_business_001",
  "nbf": "2026-04-08T10:00:00+08:00",
  "exp": "2026-04-08T11:00:00+08:00",
  "iat": "2026-04-08T10:00:00+08:00",
  "jti": "t1_001",
  "agent": {
    "agent_id": "agt_business_001",
    "agent_name": "业务数据助手",
    "app_id": "com.huawei.business.agent"
  }
}
```

## 5. TR：资源令牌

`TR` 由 `IAM` 签发，用来表达：

- 当前 Agent 代表当前用户访问资源
- 当前用户授权给这个 Agent 的权限点范围

### 5.1 TR 响应体示例

```json
{
  "code": "201",
  "message": "OK",
  "access_token": "eyJ...<TR-Payload>...signature",
  "expires_at": "2026-04-08T11:05:00+08:00",
  "expires_in": 3600,
  "token_type": "IAM_AI_SERVICE"
}
```

### 5.2 TR Payload 示例

```json
{
  "iss": "iam",
  "sub": "agt_business_001",
  "nbf": "2026-04-08T10:05:00+08:00",
  "exp": "2026-04-08T11:05:00+08:00",
  "iat": "2026-04-08T10:05:00+08:00",
  "jti": "tr_001",
  "agent": {
    "agent_id": "agt_business_001",
    "agent_name": "业务数据助手"
  },
  "agency_user": {
    "idp": "idaas",
    "user_id": "uid_001",
    "global_user_id": "z01062668",
    "username": "张建国",
    "authorizedPermissionPoints": [
      {
        "code": "erp:report:r",
        "displayNameZh": "ERP 报表的可读权限"
      },
      {
        "code": "erp:contract:r",
        "displayNameZh": "ERP 合同的可读权限"
      }
    ]
  }
}
```

### 5.3 TR 关键字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `agent.agent_id` | string | 当前 Agent ID |
| `agency_user.user_id` | string | 被代理用户唯一标识 |
| `agency_user.global_user_id` | string | 用户工号 |
| `agency_user.username` | string | 用户姓名 |
| `agency_user.authorizedPermissionPoints` | object[] | 用户授权给 Agent 的权限点对象数组 |

## 6. 运行时如何使用 TR

`TR` 不直接承载最终可调用工具列表。  
`MCP 网关` 在运行时按下面顺序做判定：

1. 从 `TR.agency_user.authorizedPermissionPoints` 读取权限点对象
2. 提取其中的 `code`
3. 确定当前工具依赖哪些权限点
4. 校验这些权限点是否都已包含在 `TR` 中
5. 再结合 Agent 策略判断当前用户是否允许使用这些功能
6. 通过后才允许调用目标工具

也就是说：

- `TR` 决定用户授权给 Agent 的权限点上限边界
- 策略决定在这些已授权权限点内，当前用户真正可用哪些功能

## 7. 当前固定结论

- `Tc` 和 `TR` 都带 `authorizedPermissionPoints`
- 每个权限点对象都至少包含 `code + displayNameZh`
- `TR` 是用户授权给 Agent 的上限边界
- 策略不能扩权
- 即使策略放行，`MCP 网关` 也必须先校验目标工具所需权限点是否已包含在 `TR` 中
