# Agent 安全令牌设计（OAuth2/JWT）

`01` 为参考方案，`02/03/04/05` 为当前正式方案。

## 1. 说明

本文描述当前版本的三类令牌：

- `Tc`：用户令牌
- `T1`：Agent 令牌
- `TR`：资源令牌

当前版本中：

- `Tc` 和 `TR` 中承载权限点对象数组的正式字段名固定为 `consented_scopes`
- 这是当前三方对接约定的 wire format 字段名，不再改为其他命名
- `consented_scopes` 的语义仍然是“权限点对象数组”
- 每个权限点对象至少包含：
  - `code`
  - `displayNameZh`
- `TR` 不是最终可调用工具列表
- `TR` 表达的是**用户授权给 Agent 的权限点上限边界**

## 2. 公共约定

### 2.1 标准 Claim

| Claim | 含义 |
| --- | --- |
| `iss` | 签发方 / 令牌发行者 |
| `sub` | 令牌代表的主体 |
| `aud` | 目标接收方 |
| `nbf` | 生效时间 |
| `exp` | 过期时间 |
| `iat` | 签发时间 |
| `jti` | JWT 全局唯一 ID |

### 2.2 令牌中的权限点对象

当前正式方案里，令牌中的权限点统一采用对象数组：

```json
"consented_scopes": [
  {
    "code": "erp:contract:r",
    "displayNameZh": "ERP 合同的可读权限"
  }
]
```

说明：

- `consented_scopes` 是当前正式字段名
- 运行时如果只需要 code，可从 `consented_scopes[].code` 提取
- `coveredPermissionPointCodes` 这类纯 code 集合只作为缓存或判定过程中的派生集合出现

这样设计的收益：

- token 自描述更强
- 审计和展示更方便
- 前端或日志不需要额外反查中文描述
- 后续如要增加 `description`、`resource`、`action` 字段也容易扩展

### 2.3 一次性凭据不是令牌

当前流程中还会出现两类一次性凭据：

| 凭据 | 用途 | 是否是令牌 |
| --- | --- | --- |
| `ticketST` | base 登录完成后，业务 Agent 后端用它向 Agent 网关换取用户信息 | 否 |
| `token_result_ticket` | 业务授权完成后，业务 Agent 后端用它向 Agent 网关换取 `TR` | 否 |

它们只是 Agent 网关内部结果的短期取件凭证，不表达用户授权范围，也不能访问资源。

统一约束：

- 单次使用。
- 短 TTL。
- 绑定 `agent_id`。
- 绑定原始 `return_url` 所属白名单。
- 交换成功后立即失效。
- 不写入 `Tc / T1 / TR`。
- 不允许替代 `consented_scopes`。
- 浏览器 URL 中只允许出现一次性凭据，不允许出现 `Tc`、`TR` 或用户信息。

## 3. Tc：用户令牌

`Tc` 由 `IDaaS` 签发，用来表达：

- 当前用户是谁
- 当前用户授权给 Agent 哪些权限点

### 3.1 接口说明

#### URI

```http
POST /oauth2/token
```

#### 请求体示例

```json
{
  "grant_type": "authorization_code",
  "code": "auth_code_001",
  "client_id": "agent_gateway_client",
  "redirect_uri": "https://agent-gateway.huawei.com/gw/auth/consent/callback"
}
```

#### 成功响应体示例

```json
{
  "code": "201",
  "message": "OK",
  "access_token": "eyJ...<Tc-Payload>...signature",
  "expires_at": "2026-04-08T11:00:00+08:00",
  "expires_in": 3600,
  "token_type": "Bearer"
}
```

### 3.2 Tc Payload 示例

```json
{
  "iss": "idaas",
  "sub": "z01062668",
  "aud": "agent-gateway",
  "nbf": "2026-04-08T10:00:00+08:00",
  "exp": "2026-04-08T11:00:00+08:00",
  "iat": "2026-04-08T10:00:00+08:00",
  "jti": "tc_001",
  "user": {
    "user_id": "z01062668",
    "uuid": "uuid~1234556",
    "username": "张建国"
  },
  "consented_scopes": [
    {
      "code": "erp:report:r",
      "displayNameZh": "ERP 报表的可读权限"
    },
    {
      "code": "erp:contract:r",
      "displayNameZh": "ERP 合同的可读权限"
    }
  ]
}
```

### 3.3 Tc 关键字段

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `user.user_id` | string | 用户工号 |
| `user.uuid` | string | 用户全局唯一标识 |
| `user.username` | string | 用户姓名 |
| `consented_scopes` | object[] | 用户已授权给 Agent 的权限点对象数组 |
| `consented_scopes[].code` | string | 权限点 code |
| `consented_scopes[].displayNameZh` | string | 权限点中文描述 |

## 4. T1：Agent 令牌

`T1` 由 `IAM` 签发，用来表达：

- 当前 Agent 是谁
- IAM 是否认可该 Agent 以程序集成身份发起资源访问

### 4.1 获取 T1 接口

#### URI

```http
POST /iam/projects/{proxy_project_id}/assume_agent_token
```

#### 请求头示例

```http
POST /iam/projects/{proxy_project_id}/assume_agent_token
Authorization: Bearer <iam_authorization_token>
Content-Type: application/json
```

说明：

- 当前版本中，请求头里的 `Authorization` 后续会通过 `IAM SDK` 获取
- 本轮实现阶段可先从配置文件读取
- 文档与代码中都应明确预留 `TODO`

#### 请求体示例

```json
{
  "data": {
    "type": "assume_agent_token",
    "attributes": {
      "delegator_account_name": "svc_ai_business_agent",
      "delegator_appid": "com.huawei.business.agent",
      "agent_id": "agt_business_001"
    }
  }
}
```

说明：

- `delegator_account_name`：委托方账号名称
- `delegator_appid`：委托方 `appid`
- `agent_id`：被委托方 `agent_id`
- 当前版本中，这三个值后续由 Agent 网关从数据库查询；本轮实现可先从配置文件读取，并保留 `TODO`

#### 成功响应体示例

```json
{
  "message": "OK",
  "code": "201",
  "enterprise": "11111111111111111111111111111111",
  "access_token": "eyJhbGciOiJSUzUxMiIsInR5cCI6IkpXVCJ9....",
  "expires_at": "2026-04-03T14:24:48+08:00",
  "expires_in": 86399,
  "token_id": "1234",
  "token_type": "IAM_AI_SERVICE",
  "expires_on": 1775197488000
}
```

### 4.2 获取 T1 成功响应字段

| 名称 | 类型 | 描述 |
| --- | --- | --- |
| `code` | String | HTTP 状态码 |
| `message` | String | 状态信息，`OK` 表示成功 |
| `enterprise` | String | 用户所属企业 ID |
| `access_token` | String | 用户 token |
| `expires_in` | String | 多少秒后过期 |
| `expires_at` | String | 具体过期时间 |
| `expires_on` | String | 过期时间戳 |
| `token_type` | String | token 类型，只返回 `IAM_AI_SERVICE` |
| `token_id` | String | token 唯一 ID |

### 4.3 T1 Payload 示例

```json
{
  "iss": "iam",
  "sub": "com.huawei.business.agent",
  "nbf": 1775024688,
  "exp": 1775197488,
  "iat": 1775111088,
  "jti": "6bfbbe98a-1d73-4bc7-a745-6efe94e9e0f9",
  "name": "svc_ai_business_agent",

  "enterprise": "11111111111111111111111111111111",

  "access_domain": "middle-secret",
```

### 4.4 T1 Payload 参数说明

| 名称 | 类型 | 描述 |
| --- | --- | --- |
| `token_id` | String | token 唯一 ID |
| `iss` | String | 签发人 / 令牌发行者 |
| `sub` | String | 主题 / 令牌代表的主体 |
| `nbf` | String | 生效时间（Unix 时间戳，秒） |
| `exp` | String | 过期时间（Unix 时间戳，秒） |
| `iat` | String | 签发时间（Unix 时间戳，秒） |
| `jti` | String | 唯一标识符（UUID 格式） |
| `name` | String | 程序集成账号名称 |
| `account_id` | String | 程序集成账号 ID，由 IAM 维护 |
| `enterprise` | String | 企业 ID |
| `account_type` | String | 账号类型，此接口只支持 `IAM_AI_SERVICE` |

## 5. TR：资源令牌

`TR` 由 `IAM` 签发，用来表达：

- 当前 Agent 代表当前用户访问资源
- 当前用户授权给这个 Agent 的权限点范围

### 5.1 获取 TR 接口

#### URI

```http
POST /iam/auth/resource-token
```

#### 请求头示例

```http
POST /iam/auth/resource-token
Authorization: Bearer <agent_token>
Content-Type: application/json
```

说明：

- 这里的 `Authorization` 就是前一步申请到的 `T1`

#### 请求体示例

```json
{
  "data": {
    "type": "resource_token",
    "attributes": {
      "user_token": "Tc"
    }
  }
}
```

#### 成功响应体示例

```json
{
  "message": "OK",
  "code": "201",
```

## 6. 运行时如何使用 TR

`TR` 不直接承载最终可调用工具列表。  
`MCP 网关` 在运行时按下面顺序做判定：

1. 从 `TR.agency_user.consented_scopes` 读取权限点对象
2. 提取其中的 `code`
3. 按当前待调用工具反查所需权限点
4. 校验这些权限点是否都已包含在 `TR` 中
5. 再结合 Agent 策略判断当前用户是否允许使用这些功能
6. 再按 `TR` 中权限点反查工具集合
7. 最后校验当前请求工具是否属于该工具集合

也就是说：

- `TR` 决定用户授权给 Agent 的权限点上限边界
- 策略决定在这些已授权权限点内，当前用户真正可用哪些功能

## 7. 当前固定结论

- `Tc` 和 `TR.agency_user` 都带 `consented_scopes`
- `consented_scopes` 是三方约定好的正式字段名，不改 wire format
- 每个权限点对象都至少包含 `code + displayNameZh`
- `TR` 是用户授权给 Agent 的上限边界
- 策略不能扩权
- 即使策略放行，`MCP 网关` 也必须先校验目标工具所需权限点是否已包含在 `TR` 中
