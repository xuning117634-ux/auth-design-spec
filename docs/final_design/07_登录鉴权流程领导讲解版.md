# 登录鉴权流程领导讲解版

本文用于会议讲解 Agent 网关登录、授权、获取 `TR`、运行时鉴权的主流程。

一句话主线：

业务 Agent 维护自己的登录会话，IDaaS 负责用户登录和授权确认，Agent 网关负责安全编排和换取令牌，MCP 网关负责最终运行时鉴权。

## 角色分工

- 用户/浏览器：负责页面访问、登录确认、授权确认，不直接接触 `Tc`、`TR` 等敏感令牌。
- 业务 Agent：维护业务侧 `site_session`，保存本地 `tr_cache`，根据用户消息判断需要调用哪些 MCP 工具。
- Agent 网关：校验 Agent 信息和回跳地址，编排 IDaaS/IAM 流程，使用一次性票据把敏感结果交给业务 Agent 后端。
- IDaaS：负责用户登录态判断、登录页、授权页、`code -> Tc`、用户信息查询。
- IAM：负责三段式令牌链路：网关自身 `agent_token`、业务 Agent `T1`、资源令牌 `TR`。
- 策略中心：维护权限点、权限点与工具映射、Agent 策略。
- MCP 网关：运行时校验 `TR.agency_user.consented_scopes`、Agent 策略和工具覆盖关系。
- MCP Server：真正执行业务工具调用。

## 简化版图一：登录建会话

```mermaid
sequenceDiagram
    actor U as 用户/浏览器
    participant BA as 业务 Agent
    participant AG as Agent 网关
    participant ID as IDaaS

    U->>BA: 访问业务 Agent
    BA->>BA: 检查业务侧登录态
    BA-->>U: 引导到 Agent 网关登录

    U->>AG: 发起登录请求
    AG->>AG: 校验 Agent 和回跳地址
    AG-->>U: 引导到 IDaaS 登录

    U->>ID: 完成登录或复用登录态
    ID-->>AG: 返回登录授权码
    AG->>AG: 生成一次性登录票据
    AG-->>BA: 通过浏览器回跳传递一次性票据

    BA->>AG: 后端用一次性票据换用户信息
    AG->>ID: 换取用户信息
    ID-->>AG: 返回用户信息
    AG-->>BA: 返回用户信息
    BA->>BA: 创建业务侧会话
    BA-->>U: 展示已登录页面
```

## 简化版图二：授权并换取 TR

```mermaid
sequenceDiagram
    actor U as 用户/浏览器
    participant BA as 业务 Agent
    participant AG as Agent 网关
    participant PC as 策略中心
    participant ID as IDaaS
    participant IAM as IAM

    U->>BA: 发起需要工具能力的问题
    BA->>BA: 判断需要哪些工具
    BA->>BA: 检查本地是否已有可用 TR
    BA->>AG: 无 TR 时申请授权

    AG->>AG: 校验 Agent 和工具范围
    AG->>PC: 根据工具查询所需权限点
    PC-->>AG: 返回权限点
    AG->>AG: 创建授权事务
    AG-->>BA: 返回授权跳转地址

    BA-->>U: 浏览器跳转授权页
    U->>ID: 用户确认授权
    ID-->>AG: 返回授权码

    AG->>ID: 换取用户授权结果
    ID-->>AG: 返回用户实际同意的权限点
    AG->>IAM: 获取网关自身令牌
    IAM-->>AG: 返回网关令牌
    AG->>IAM: 获取业务 Agent 令牌
    IAM-->>AG: 返回业务 Agent 令牌
    AG->>IAM: 换取资源令牌 TR
    IAM-->>AG: 返回 TR

    AG->>AG: 生成一次性 TR 结果票据
    AG-->>BA: 通过浏览器回跳传递一次性票据
    BA->>AG: 后端用一次性票据换 TR
    AG-->>BA: 返回 TR
    BA->>BA: 校验用户一致并缓存 TR
    BA-->>U: 恢复原问题并继续执行
```

## 简化版图三：MCP 运行时鉴权

```mermaid
sequenceDiagram
    participant BA as 业务 Agent
    participant MCP as MCP 网关
    participant PC as 策略中心
    participant MS as MCP Server

    BA->>MCP: 携带 TR 和目标工具发起调用
    MCP->>PC: 查询目标工具需要哪些权限点
    PC-->>MCP: 返回工具所需权限点
    MCP->>MCP: 校验权限点是否在 TR 授权范围内

    MCP->>PC: 查询当前 Agent 的策略
    PC-->>MCP: 返回策略
    MCP->>MCP: 判断当前用户是否通过策略

    MCP->>PC: 查询 TR 权限点覆盖哪些工具
    PC-->>MCP: 返回工具集合
    MCP->>MCP: 校验目标工具是否被 TR 覆盖

    MCP->>MS: 鉴权通过后调用真实工具
    MS-->>MCP: 返回工具结果
    MCP-->>BA: 返回结果
```

## 图一：登录建会话

这张图只讲“用户如何登录业务 Agent”。核心点是：浏览器只拿到一次性 `ticketST`，业务 Agent 后端再用它换用户信息，最终由业务 Agent 创建自己的 `site_session`。

```mermaid
sequenceDiagram
    actor U as 用户/浏览器
    participant BA as 业务 Agent
    participant AG as Agent 网关
    participant ID as IDaaS

    U->>BA: 1. 访问 /agent
    BA->>BA: 2. 检查业务侧 site_session
    BA-->>U: 3. 未登录，302 到 /gw/auth/login

    U->>AG: 4. GET /gw/auth/login?agent_id&return_url&state
    AG->>AG: 5. 校验 Agent 信息和 return_url 白名单
    AG->>AG: 6. 创建 pending_base_login
    AG-->>U: 7. 302 到 IDaaS authorize，scope=base.profile

    U->>ID: 8. 用户登录或复用 IDaaS 登录态
    ID-->>U: 9. 302 回 /gw/auth/base/callback?code&state

    U->>AG: 10. GET /gw/auth/base/callback?code&state
    AG->>AG: 11. 保存 code，生成一次性 ticketST
    AG-->>U: 12. 302 回业务 Agent return_url?ticketST&state

    U->>BA: 13. GET /agent?ticketST&state
    BA->>AG: 14. POST /gw/auth/ticket/exchange {agent_id,ticketST}
    AG->>ID: 15. code 换 Tc 短 access_token
    ID-->>AG: 16. 返回 Tc
    AG->>ID: 17. 用 Tc 获取 userinfo
    ID-->>AG: 18. 返回用户信息
    AG-->>BA: 19. 返回用户信息
    BA->>BA: 20. 创建业务侧 site_session
    BA-->>U: 21. 展示已登录页面
```

## 图二：授权并换取 TR

这张图只讲“用户要调用工具时，业务 Agent 如何拿到 `TR`”。核心点是：用户同意授权后，浏览器也不接触 `TR`，只拿到一次性 `token_result_ticket`。

```mermaid
sequenceDiagram
    actor U as 用户/浏览器
    participant BA as 业务 Agent
    participant AG as Agent 网关
    participant PC as 策略中心
    participant ID as IDaaS
    participant IAM as IAM

    U->>BA: 1. 输入问题，例如“帮我看一下 ERP 合同”
    BA->>BA: 2. 识别 required_tools
    BA->>BA: 3. 检查本地 tr_cache
    BA->>AG: 4. 无 TR，POST /gw/token/resource-token {agentId,requiredTools,returnUrl,state}

    AG->>AG: 5. 校验 Agent 信息、return_url、required_tools
    AG->>PC: 6. resolve-by-tools，工具反查权限点
    PC-->>AG: 7. 返回 requiredPermissionPointCodes
    AG->>AG: 8. 创建 pending_auth_transaction
    AG-->>BA: 9. 返回 REDIRECT_REQUIRED + redirect_url + request_id

    BA-->>U: 10. 保存原始消息到 sessionStorage，跳转 redirect_url
    U->>AG: 11. GET /gw/auth/authorize?request_id
    AG->>AG: 12. 找到授权事务，拼接 scope
    AG-->>U: 13. 302 到 IDaaS authorize，scope=权限点 code 空格拼接

    U->>ID: 14. 用户确认授权
    ID-->>U: 15. 302 回 /gw/auth/consent/callback?code&state

    U->>AG: 16. GET /gw/auth/consent/callback?code&state
    AG->>ID: 17. code 换 Tc
    ID-->>AG: 18. 返回 Tc，Tc 中包含 consented_scopes
    AG->>AG: 19. 解析 Tc.consented_scopes

    AG->>IAM: 20. 用 Agent 网关 AI 账号获取 gateway agent_token
    IAM-->>AG: 21. 返回 gateway agent_token
    AG->>IAM: 22. 用 gateway agent_token 获取业务 Agent T1
    Note over AG,IAM: delegator_account_name = Agent_ + agentId<br/>delegator_appid = appId
    IAM-->>AG: 23. 返回 T1
    AG->>IAM: 24. 用 T1 + Tc 换 TR
    IAM-->>AG: 25. 返回 TR

    AG->>AG: 26. 生成一次性 token_result_ticket
    AG-->>U: 27. 302 回业务 Agent return_url?token_result_ticket&request_id&state

    U->>BA: 28. GET /agent?token_result_ticket&request_id&state
    BA->>AG: 29. POST /gw/token/result/exchange {agent_id,request_id,token_result_ticket}
    AG-->>BA: 30. 返回 TR、consented_scopes、agency_user
    BA->>BA: 31. 校验 TR 用户与当前 site_session 用户一致
    BA->>BA: 32. 写入本地 tr_cache
    BA-->>U: 33. 从 sessionStorage 恢复原消息并重发
```

## 图三：MCP 运行时鉴权

这张图只讲“拿到 `TR` 后，MCP 网关如何判断工具能不能真正调用”。核心点是：运行时不是只看 `TR`，还要结合 Agent 策略和工具映射做最终校验。

```mermaid
sequenceDiagram
    participant BA as 业务 Agent
    participant MCP as MCP 网关
    participant PC as 策略中心
    participant MS as MCP Server

    BA->>MCP: 1. 携带 TR + agentId + toolId 调用 MCP 网关
    MCP->>PC: 2. resolve-by-tools，查询当前 toolId 依赖哪些权限点
    PC-->>MCP: 3. 返回当前工具所需权限点
    MCP->>MCP: 4. 校验权限点是否都在 TR.agency_user.consented_scopes 中

    MCP->>PC: 5. agent-strategies/query，查询 agentId 下相关权限点策略
    PC-->>MCP: 6. 返回 Agent 策略
    MCP->>MCP: 7. 执行策略判断：DENY 优先，无策略默认允许，有 PERMIT 必须命中

    MCP->>PC: 8. resolve-by-codes，用 TR 权限点反查覆盖工具集合
    PC-->>MCP: 9. 返回 TR 覆盖的工具集合
    MCP->>MCP: 10. 校验当前 toolId 在 TR 覆盖工具集合中

    MCP->>MS: 11. 校验通过，转发到 MCP Server
    MS-->>MCP: 12. 返回工具结果
    MCP-->>BA: 13. 返回 MCP 结果
```

## 讲解重点

### 1. 为什么要引入一次性票据

浏览器 URL 中不出现 `Tc`、`TR`、用户信息、`gw_session_token` 这类敏感数据。

登录完成后，Agent 网关只把一次性 `ticketST` 放到 URL 中。业务 Agent 后端拿 `ticketST` 向 Agent 网关换用户信息，然后创建自己的 `site_session`。

授权完成后，Agent 网关只把一次性 `token_result_ticket` 放到 URL 中。业务 Agent 后端拿 `token_result_ticket` 向 Agent 网关换 `TR`，再写入自己的 `tr_cache`。

### 2. 为什么 Agent 网关不维护用户长期登录态

用户是否已经登录，由 IDaaS 判断。

业务 Agent 是否已经有自己的登录会话，由业务 Agent 的 `site_session` 判断。

Agent 网关只维护短事务和一次性票据，不长期保存用户登录态，也不替业务 Agent 管理会话。

### 3. TR 和策略分别解决什么问题

`TR` 是用户授权给 Agent 的权限上限，表示用户同意这个 Agent 可以访问哪些权限点。

Agent 策略是 Agent 应用所有者对不同用户开放功能的二次控制，例如某些用户不能使用某个权限点对应的能力。

最终 MCP 工具能不能调用，需要同时满足：

- 当前工具依赖的权限点在 `TR.agency_user.consented_scopes` 中。
- 当前用户通过 Agent 策略判断。
- 当前工具属于 `TR` 权限点反查得到的工具集合。

### 4. IAM 三段式令牌关系

第一段：Agent 网关用自己的 AI 账号获取 `gateway agent_token`。

第二段：Agent 网关用 `gateway agent_token` 获取业务 Agent 的 `T1`。

第三段：Agent 网关用业务 Agent 的 `T1` 和用户授权得到的 `Tc` 换取 `TR`。

其中业务 Agent 的 IAM AI 账号名规则为：

```text
delegator_account_name = Agent_ + agentId
delegator_appid = appId
```

## 汇报版总结

这套方案把登录、授权、令牌、运行时鉴权拆成清晰边界：

- 登录态归业务 Agent 和 IDaaS 管。
- 授权编排归 Agent 网关管。
- 权限点和策略归策略中心管。
- 最终工具调用裁决归 MCP 网关管。

核心安全收益是：敏感令牌不进浏览器 URL，业务 Agent 后端通过一次性票据换取结果；运行时不只看 `TR`，还会结合 Agent 策略和工具映射做最终校验。
