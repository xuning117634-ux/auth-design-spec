# 策略中心设计

`01` 为参考方案，`02/03/04/05` 为当前正式方案。

策略中心同时维护两类核心对象：权限点定义，以及 Agent 维度的附加控制策略；`TR` 中携带的是权限点 code，表达用户授权给 Agent 的可访问上限范围；`MCP 网关` 在运行时再结合 Agent 策略限制不同用户在此 Agent 下的权限。

## 1. 目标

当前正式方案中：

- `TR` 中携带的是**权限点 code**
- `TR` 表达的是**用户授权给 Agent 的最大访问范围**
- 权限点由 `MCP 网关` 定义并上报策略中心
- 策略由 `Agent 网关` 创建并上报策略中心
- 当前版本中，一个策略只绑定**一个权限点**
- 策略 = 权限点 + 条件 + `effect`
- 策略表达的是**Agent 应用所有者对不同用户可用功能的二次控制**
- 运行时策略判断由 `MCP 网关` 执行
- 只要当前请求依赖的任一权限点最终被拒绝，且当前请求中调用的 `MCP tool` 属于该被拒绝权限点，则当前请求直接失败

## 2. 核心术语

### 2.1 权限点

权限点是 `TR` 中携带的授权对象，由 `MCP 网关` 定义并上报策略中心。

权限点 code 组成规则：

```text
<归属应用简写>:<resource描述>:<action>
```

例如：

```text
erp:report:r
```

当前版本中，权限点由 `MCP 网关` 负责关联对应工具，并将“权限点与工具的绑定关系”上传到策略中心。

### 2.2 策略

策略是 Agent 维度对某个权限点附加的控制规则。  
当前版本中，一个策略只作用于一个权限点；策略不是新的授权对象，而是对权限点生效条件的补充控制。

可以把当前模型理解成两层边界：

- `TR`：用户授权给 Agent 的上限边界
- 策略：Agent 应用所有者对用户可用功能的二次控制

## 3. 权限点模型

每个权限点至少包含以下字段：

- `permissionPointCode`
- `displayNameZh`
- `description`
- `boundTools`
- `status`

示例：

```json
{
  "permissionPointCode": "erp:report:r",
  "displayNameZh": "ERP 报表的可读权限",
  "description": "允许读取 ERP 报表数据",
  "boundTools": [
    "mcp:financial-report-server/query_monthly_report",
    "mcp:financial-report-server/list_report_categories"
  ],
  "status": "ACTIVE"
}
```

当前版本只明确以下结论：

- 权限点由 `MCP 网关` 创建
- 策略中心维护权限点与工具的绑定关系
- `TR` 中携带的 code 就是权限点 code

## 4. 策略模型

当前版本中，一个策略只绑定一个权限点。

每个策略至少包含以下字段：

- `strategyId`
- `agentId`
- `permissionPointCode`
- `conditions`
- `effect`
- `status`

示例：

```json
{
  "strategyId": "stg_001",
  "agentId": "agt_finance_assistant",
  "permissionPointCode": "erp:report:r",
  "conditions": {
    "field": "subject.user_id",
    "operator": "in",
    "values": ["u_001", "u_002"]
  },
  "effect": "DENY",
  "status": "ACTIVE"
}
```

当前版本中，策略条件只支持用户 ID 条件：

- `field` 固定为 `subject.user_id`
- `operator` 固定支持：
  - `equals`
  - `in`

当前版本先把策略能力收敛到“指定哪些用户对某个权限点允许或禁止”，不把策略中心扩展成通用规则引擎。

## 5. 职责划分

### 5.1 Agent 网关

- 负责 `required_tools -> permissionPointCodes`
- 负责把用户授权后的权限点 code 放入 `TR`
- 不执行运行时策略判断

### 5.2 策略中心

- 维护权限点定义
- 维护权限点与工具的绑定关系
- 维护 Agent 维度策略
- 提供映射和策略查询接口
- 不执行最终访问裁决

### 5.3 MCP 网关

- 从 `TR` 中读取：
  - `agent_id`
  - 用户信息
  - 权限点 code
- 运行时查询策略中心
- 对策略做命中判断
- 决定当前请求中的目标工具是否允许执行

## 6. 运行时裁决规则

### 6.1 前提

所有策略判断都只在 **`TR` 已经授予该权限点** 的前提下生效：

- `TR` 是基础授权
- 策略是在 `TR` 基础上的进一步限制
- 策略不能扩权
- 即使某条策略最终放行，`MCP 网关` 也仍然必须先校验当前请求对应工具所需的权限点是否已经在 `TR` 授权范围内

也就是说：

- `TR` 决定当前用户愿意让这个 Agent 访问哪些权限点
- 策略决定在这些已授权权限点之内，Agent 对当前用户实际开放哪些功能

### 6.2 判定顺序

对**当前请求实际调用的 `MCP tool` 所依赖的权限点**，`MCP 网关` 按下面顺序判断：

1. 先校验该 `permissionPointCode` 是否已经在 `TR` 的授权范围内
   - 如果不在，则当前请求直接失败
2. 如果该 `agentId + permissionPointCode` **没有任何策略**
   - 默认允许该权限点生效
3. 如果存在命中的 `DENY` 策略
   - 直接拒绝该权限点
4. 否则，如果该权限点存在 `PERMIT` 策略
   - 必须至少命中一条 `PERMIT` 才允许
   - 一条都没命中则拒绝该权限点

### 6.3 请求级结果

当前请求只对**本次实际调用的 `MCP tool` 所依赖的权限点**做判断。

只要当前请求依赖的任一权限点最终被拒绝，且当前请求中调用的 `MCP tool` 属于该被拒绝权限点：

- 当前请求直接失败
- 返回没权限

与当前请求无关的其他权限点，不参与本次裁决，也不影响本次请求。

规则总结：

```text
按本次实际调用工具做权限判定；只要当前请求依赖的任一权限点最终被拒绝，且当前请求中调用的 MCP tool 属于该被拒绝权限点，则当前请求直接失败。
```

换句话说，当前工具可调用的前提是：

```text
当前请求所需权限点已经包含在 TR 中，并且通过了 Agent 策略判断。
```

## 7. 协作流程说明

### 7.1 Agent 网关侧

- 业务 Agent 上传 `required_tools`
- `Agent 网关` 向策略中心查询需要申请哪些权限点
- 用户完成授权
- `TR` 中携带对应权限点 code

### 7.2 MCP 网关侧

- `MCP 网关` 收到 `TR` 和目标工具请求
- 先从 `TR` 中解析当前用户信息和权限点 code
- 再确定本次实际调用的 `MCP tool` 依赖哪些权限点
- 先校验这些权限点是否已经在 `TR` 授权范围内
- 再从策略中心查询当前 Agent 对这些权限点的策略
- 按策略规则做运行时判定
- 如果本次实际调用工具依赖的权限点被拒绝，则当前请求失败

## 8. 策略中心接口建议

### 8.1 MCP 网关写权限点

```http
PUT /internal/v1/permission-points/batch-upsert
```

用途：

- 批量新增或更新权限点定义
- 同步权限点与工具的绑定关系

### 8.2 Agent 网关写策略

```http
PUT /internal/v1/agent-strategies/batch-upsert
```

用途：

- 批量新增或更新 Agent 维度策略

### 8.3 Agent 网关按工具反查权限点

```http
POST /internal/v1/permission-points/resolve-by-tools
```

输入：

- `agentId`
- `requiredTools`

输出：

- `permissionPointCodes`
- `permissionPoints`

### 8.4 MCP 网关按 Agent + 权限点查询策略

```http
POST /internal/v1/agent-strategies/query
```

输入：

- `agentId`
- `permissionPointCodes`

输出：

- `strategies`

## 9. 当前版本关键约束

- `TR` 中携带的是权限点 code
- 权限点由 `MCP 网关` 定义并上报策略中心
- 策略由 `Agent 网关` 创建并上报策略中心
- 当前版本中，一个策略只绑定一个权限点
- 当前策略条件字段只支持 `subject.user_id`
- `DENY` 优先
- 没有策略时默认允许
- 存在 `PERMIT` 策略时必须命中至少一条
- 运行时判断由 `MCP 网关` 执行
- `TR` 是用户授权给 Agent 的上限边界
- 策略是 Agent 应用所有者对不同用户可用功能的二次控制
- 即使策略放行，`MCP 网关` 也必须先校验当前请求工具所需权限点是否已在 `TR` 授权范围内
- 只要当前请求依赖的任一权限点最终被拒绝，且当前请求中调用的 `MCP tool` 属于该被拒绝权限点，则当前请求直接失败
- 与当前请求无关的权限点不影响本次请求
- 策略不能扩权，只能在 `TR` 已授予范围内做限制
