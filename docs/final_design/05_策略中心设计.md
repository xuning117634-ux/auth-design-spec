# 策略中心设计

策略中心同时维护三类核心对象：权限点定义、Agent 订阅权限点快照，以及 Agent 维度的附加控制策略；`TR` 中携带的是权限点对象，MCP 网关在运行时结合 Agent 策略决定当前权限点是否真正生效。

## 1. 设计主线

- `TR` 表达用户授权给 Agent 的最大访问范围。
- `TR.agency_user.consented_scopes` 是三方约定字段，格式为权限点 code 字符串数组。
- `TR.scope` 是预留字段，不作为运行时鉴权依据。
- 权限点由 `MCP 网关` 定义并上报策略中心。
- `permissionPointCode` 全系统唯一，且不能包含英文逗号 `,`。
- 权限点上报时必须携带 `enterprise` 和 `appId`。
- `enterprise` 是企业隔离边界，`appId` 是权限点归属应用。
- Agent 订阅权限点快照由 `Agent 网关` 上报策略中心。
- Agent 策略由 `Agent 网关` 创建并上报策略中心。
- MCP 网关运行时负责 `TR` 范围校验和 Agent 策略判断。

## 2. 权限点模型

权限点是 `TR` 中携带的授权对象，也是 MCP 工具运行时校验的最小授权点。

权限点 code 由 `MCP 网关` 创建，建议组成规则：

```text
<归属应用简写>:<resource描述>:<action>
```

示例：

```text
erp:contract:r
```

权限点结构：

```json
{
  "permissionPointCode": "erp:contract:r",
  "enterprise": "ent_001",
  "appId": "erp",
  "displayNameZh": "ERP 合同的可读权限",
  "description": "允许读取 ERP 合同数据",
  "boundTools": [
    {
      "toolId": "mcp:contract-server/get_contract",
      "displayNameZh": "查询合同详情"
    }
  ],
  "status": "ACTIVE"
}
```

规则：

- `permissionPointCode` 全系统唯一。
- `permissionPointCode` 不能包含英文逗号 `,`，因为 Agent 订阅快照会在数据库中用逗号拼接保存。
- `enterprise` 和 `appId` 用于归属、隔离和查询，不参与唯一键。
- `boundTools` 表示权限点与 MCP tool 的绑定关系。
- 权限点状态采用 `ACTIVE / INACTIVE`。
- `ACTIVE` 权限点参与运行时解析。
- `INACTIVE` 权限点保留在存储中，但不参与运行时解析。

## 3. Agent 订阅权限点快照

Agent 订阅权限点快照用于记录某个 Agent 在某个企业下当前订阅了哪些权限点。

当前该快照只作为注册面和管理面存储数据，不参与 MCP 网关运行时裁决。

快照结构：

```json
{
  "agentId": "agt_business_001",
  "enterprise": "ent_001",
  "permissionPointCodes": [
    "erp:contract:r",
    "erp:report:r"
  ]
}
```

规则：

- 一个 `agentId + enterprise` 只保存一条订阅快照。
- `permissionPointCodes` 表示当前完整订阅集合。
- 取消订阅某个权限点时，重新上报取消后的剩余完整集合。
- 清空订阅时，上报空数组。
- 数据库中使用 `permission_point_codes` 字符串字段保存，用英文逗号拼接。
- 服务端负责去空、去重、稳定排序和权限点存在性校验。
- `permissionPointCodes[]` 中的每个 code 必须已存在于当前 `enterprise` 下的权限点表。
- `permissionPointCodes[]` 中的每个 code 不能包含英文逗号 `,`。

## 4. Agent 策略模型

策略是 Agent 应用所有者对不同用户可用功能的二次控制。

当前版本中，一个策略只作用于一个权限点；策略不是新的授权对象，而是对权限点生效条件的补充控制。

策略结构：

```json
{
  "strategyId": "stg_001",
  "agentId": "agt_finance_assistant",
  "permissionPointCode": "erp:contract:r",
  "conditions": {
    "field": "subject.user_id",
    "operator": "in",
    "values": ["u_001", "u_002"]
  },
  "effect": "DENY",
  "status": "ACTIVE"
}
```

规则：

- 当前条件字段只支持 `subject.user_id`。
- 当前运算符只支持 `equals` 和 `in`。
- `effect` 支持 `PERMIT / DENY`。
- `DENY` 优先。
- 策略采用 `ACTIVE / INACTIVE`。
- 当前版本先把策略能力收敛到“指定哪些用户对某个权限点允许或禁止”，不把策略中心扩展成通用规则引擎。

## 5. 职责边界

Agent 网关：

- 负责 `required_tools -> permissionPointCodes`。
- 负责把用户授权后的权限点 code 数组放入 `TR.agency_user.consented_scopes`。
- 负责在 Agent 注册或订阅权限点时，上报 Agent 订阅权限点快照。
- 不执行运行时策略判断。

策略中心：

- 维护权限点定义。
- 维护权限点与工具绑定关系。
- 维护 Agent 订阅权限点快照。
- 维护 Agent 维度策略。
- 提供映射和策略查询接口。
- 不执行最终访问裁决。

MCP 网关：

- 从 `TR.aud` 读取业务 Agent ID，从 `TR.agency_user.user` JSON 字符串读取用户 `uid`，并读取 `consented_scopes`。
- 查询当前工具依赖的权限点。
- 校验当前工具依赖的权限点是否在 `TR.agency_user.consented_scopes` 中。
- 查询当前 `agentId + permissionPointCodes` 的 Agent 策略。
- 执行运行时策略判定。
- 再按 `TR` 中权限点反查工具集合，并确认当前工具在集合内。

## 6. 运行时裁决规则

所有策略判断都只在 `TR` 已经授予该权限点的前提下生效：

- `TR` 是基础授权。
- 策略是在 `TR` 基础上的进一步限制。
- 策略不能扩权。

对当前请求实际调用的 MCP 工具所依赖的权限点，MCP 网关按下面顺序判断：

1. 先按当前工具调用 `resolve-by-tools`，得到该工具所需权限点。
2. 校验这些权限点是否都包含在 `TR.agency_user.consented_scopes` 中。
3. 如果不在 `TR` 中，当前请求直接失败。
4. 查询当前 `agentId + permissionPointCodes` 的 Agent 策略。
5. 如果该 `agentId + permissionPointCode` 没有任何策略，默认允许该权限点生效。
6. 如果存在命中的 `DENY` 策略，直接拒绝该权限点。
7. 否则，如果该权限点存在 `PERMIT` 策略，必须至少命中一条 `PERMIT` 才允许；一条都没命中则拒绝。
8. 再按 `TR.agency_user.consented_scopes` 调用 `resolve-by-codes`，确认当前工具在 `TR` 可覆盖工具集合内。

当前版本采用“按本次实际调用工具做权限判定”的原则。只要当前工具所需权限点不通过策略，或当前请求工具不在 `TR` 反查出的工具集合内，则当前请求直接失败。

## 7. 策略中心接口

MCP 网关写权限点：

```http
PUT /internal/v1/permission-points/batch-upsert
```

管理台或 Agent 网关查询权限点：

```http
POST /internal/v1/permission-points/query
```

Agent 网关写 Agent 订阅权限点快照：

```http
PUT /internal/v1/agent-permission-points/batch-upsert
```

Agent 网关写策略：

```http
PUT /internal/v1/agent-strategies/batch-upsert
```

Agent 网关按工具反查权限点：

```http
POST /internal/v1/permission-points/resolve-by-tools
```

MCP 网关按 Agent + 权限点查询策略：

```http
POST /internal/v1/agent-strategies/query
```

MCP 网关按权限点反查工具：

```http
POST /internal/v1/permission-points/resolve-by-codes
```

## 8. 当前版本关键约束

- `TR` 中携带的是权限点 code 字符串数组。
- 权限点由 `MCP 网关` 定义并上报策略中心。
- `permissionPointCode` 全系统唯一，且不能包含英文逗号 `,`。
- 权限点上报时必须携带 `enterprise` 和 `appId`。
- Agent 订阅权限点快照由 `Agent 网关` 上报策略中心。
- Agent 订阅权限点快照使用全量覆盖语义。
- 策略由 `Agent 网关` 创建并上报策略中心。
- 一个策略只绑定一个权限点。
- 当前策略条件字段只支持 `subject.user_id`。
- `DENY` 优先。
- 没有策略时默认允许。
- 存在 `PERMIT` 策略时必须命中至少一条。
- 运行时判断由 `MCP 网关` 执行。
- 与当前请求无关的权限点不影响本次请求。
- 策略不能扩权，只能在 `TR` 已授予范围内做限制。
