## 2. 5.1 base 登录阶段逐箭头说明

对应 [02_引入Agent网关版方案.md](/D:/IDEA_Project/init_env/auth-design-spec/docs/final_design/02_引入Agent网关版方案.md) 的 `5.1 base 登录阶段`。

### 2.1 `用户 -> 业务 Agent : GET /agent`

含义：
- 用户第一次打开业务 Agent 页面入口。
- 此时还不知道用户有没有在业务 Agent 侧建立本地站点登录态。

参考请求：

```http
GET /agent
Cookie: site_session_id=site_789
```

说明：
- 如果是首次访问，通常没有 `site_session_id`。
- 如果是已经访问过，浏览器可能已经带上了 `site_session_id`。

### 2.2 `业务 Agent -> 业务 Agent : 检查 site_session_id`

含义：
- 业务 Agent 先检查自己是否已经有本地站点登录态。
- 这一步只看业务 Agent 自己的会话，不看网关会话。

参考数据：

```text
读取业务 Agent 域名下 Cookie：
site_session_id = site_789
```

可能结果：
- 命中 `site_session`：说明业务 Agent 自己已经认识这个用户
- 未命中：说明需要走网关登录流程

### 2.3 `业务 Agent -> 用户 : 302 到 /gw/auth/login`

含义：
- 如果业务 Agent 没有本地站点登录态，就把浏览器送到 Agent 网关。
- 这一步是把登录责任交给网关，不是业务 Agent 自己去接 OAuth。

参考响应：

```http
302 Location:
https://agent-gateway.huawei.com/gw/auth/login
  ?agent_id=agt_business_001
  &return_url=https%3A%2F%2Fbusiness-agent.huawei.com%2Fagent
  &state=st_login_001
```

说明：
- `agent_id` 表示当前是谁在发起登录流程
- `return_url` 表示流程结束后要回到业务 Agent 的哪个页面
- `state` 是业务 Agent 自己的外层状态，便于回跳后恢复页面上下文

### 2.4 `用户 -> Agent 网关 : GET /gw/auth/login`

含义：
- 浏览器正式进入网关的 base 登录入口。
- 网关开始接管整个 base 登录过程。

参考请求：

```http
GET /gw/auth/login?agent_id=agt_business_001&return_url=...&state=st_login_001
```

### 2.5 `Agent 网关 -> 用户 : 302 到 IDaaS /authorize?scope=base`

含义：
- 网关校验 `agent_id`、`return_url` 后，创建临时登录事务 `pending_base_login`。
- 随后网关把浏览器送去 IDaaS 完成 base 登录。

参考响应：

```http
302 Location:
https://idaas.huawei.com/oauth2/authorize
  ?response_type=code
  &client_id=gw_client_001
  &redirect_uri=https%3A%2F%2Fagent-gateway.huawei.com%2Fgw%2Fauth%2Fbase%2Fcallback
  &scope=base
  &state=gwst_login_001
```

说明：
- 这里的 `state=gwst_login_001` 是网关内部状态，不是业务 Agent 的外层 `state`
- `redirect_uri` 指向网关自己的 callback，而不是业务 Agent

### 2.6 `用户 -> IDaaS : 登录`

含义：
- 用户在 IDaaS 页面输入工号和密码，完成统一身份认证。

参考数据：

```text
user_id = z01062668
username = 张三
password = ********
```

### 2.7 `IDaaS -> 用户 : 回调 /gw/auth/base/callback`

含义：
- IDaaS 登录完成后，先把浏览器回调到网关，而不是直接回业务 Agent。
- 因为网关才是这套架构里的统一认证编排方。

参考响应：

```http
302 Location:
https://agent-gateway.huawei.com/gw/auth/base/callback
  ?code=code_base_001
  &state=gwst_login_001
```

### 2.8 `用户 -> Agent 网关 : 回网关`

含义：
- 浏览器跟随上一步重定向，把 `code + state` 带回网关。
- 网关据此继续完成后续换 token 和建立网关会话。

参考请求：

```http
GET /gw/auth/base/callback?code=code_base_001&state=gwst_login_001
```

### 2.9 `Agent 网关 -> IDaaS : POST /oauth2/token`

含义：
- 网关用授权码向 IDaaS 换取 base 登录结果。
- 这一跳的目标是确认“当前登录用户是谁”。

参考请求体：

```json
{
  "grant_type": "authorization_code",
  "code": "code_base_001",
  "redirect_uri": "https://agent-gateway.huawei.com/gw/auth/base/callback",
  "client_id": "gw_client_001",
  "client_secret": "******"
}
```

### 2.10 `IDaaS -> Agent 网关 : 返回 base 登录结果`

含义：
- 网关拿到当前登录用户的基础身份信息。
- 这一步之后，网关已经知道“是谁完成了 base 登录”。

参考响应：

```json
{
  "user_id": "z01062668",
  "username": "张三"
}
```

### 2.11 `Agent 网关 -> 用户 : 带 gw_session_token 回业务 Agent`

含义：
- 网关先创建自己的登录会话 `gw_session`。
- 然后网关生成 `gw_session_token`，通过 `return_url` 把它交给业务 Agent。
- 这是网关和业务 Agent 之间第一次建立“同一个登录用户”的桥接关系。

参考处理结果：

```text
gw_session_id = gws_123

gw_session = {
  gw_session_id: gws_123,
  user_id: z01062668,
  username: 张三
}

gw_session_token = gwst_abc_xyz
gw_session_token -> gws_123
```

参考响应：

```http
Set-Cookie: gw_session_id=gws_123; Path=/; HttpOnly; SameSite=Lax
302 Location:
https://business-agent.huawei.com/agent
  ?gw_session_token=gwst_abc_xyz
  &user_id=z01062668
  &username=%E5%BC%A0%E4%B8%89
  &state=st_login_001
```

说明：
- `gw_session_id` 只给网关自己用
- `gw_session_token` 是网关发给业务 Agent 的桥接凭证

### 2.12 `用户 -> 业务 Agent : 回原页面`

含义：
- 浏览器跟随网关的重定向回到业务 Agent 页面。
- 此时业务 Agent 收到了 `gw_session_token` 和用户基础信息。

参考请求：

```http
GET /agent?gw_session_token=gwst_abc_xyz&user_id=z01062668&username=张三&state=st_login_001
```

### 2.13 `业务 Agent -> 业务 Agent : 建立 site_session`

含义：
- 业务 Agent 现在创建自己的本地站点会话 `site_session`。
- 这一步把业务 Agent 的本地会话和网关已登录用户正式绑定起来。

参考数据：

```text
site_session_id = site_789

site_session = {
  site_session_id: site_789,
  gw_session_token: gwst_abc_xyz,
  user_id: z01062668,
  username: 张三
}
```

参考响应：

```http
Set-Cookie: site_session_id=site_789; Path=/; HttpOnly; SameSite=Lax
```

说明：
- `site_session_id` 定位业务 Agent 本地会话
- `gw_session_token` 让业务 Agent 后续能找回网关侧已登录用户

### 2.14 `已有站点登录态 -> 直接返回聊天页`

含义：
- 如果步骤 `2.2` 就已经命中了 `site_session`，则不需要再走网关登录。
- 业务 Agent 可以直接返回聊天页或业务主页面。

## 3. 5.2 业务授权 + 获取 TR 阶段逐箭头说明

对应 [02_引入Agent网关版方案.md](/D:/IDEA_Project/init_env/auth-design-spec/docs/final_design/02_引入Agent网关版方案.md) 的 `5.2 业务授权 + 获取 TR 阶段`。

### 3.1 `用户 -> 业务 Agent : POST /chat/send`

含义：
- 用户开始真正发起业务请求。
- 这一步通常是聊天消息、表单提交、按钮触发等。

参考请求：

```json
{
  "message": "帮我看一下这个合同"
}
```

### 3.2 `业务 Agent -> 业务 Agent : 判断本次请求需要哪些 MCP tools`

含义：
- 业务 Agent 不直接理解权限点，只先判断要调用哪些工具。
- 这是“业务意图 -> 工具”的映射阶段。

参考数据：

```text
required_tools = [
  "mcp:contract-server/get_contract"
]
```

### 3.3 `业务 Agent -> 业务 Agent : 检查本地 TR 缓存`

含义：
- 业务 Agent 先看本地 `tr_cache` 里是否已经有覆盖当前工具集合的有效 `TR`。
- 如果有，就不需要再向网关申请。

参考缓存模型：

```text
key = site_session_id + agent_id

value = {
  current_tr,
  covered_tools,
  coveredPermissionPointCodes,
  expires_at
}
```

## 3.4 本地已有可用 TR 分支

### 3.4.1 `业务 Agent -> MCP 网关 : 带 TR 调 MCP`

含义：
- 如果本地已经有可用 `TR`，业务 Agent 直接带着 `TR` 调 MCP 网关。

参考请求：

```http
POST /mcp/v1/tools/invoke
Authorization: Bearer <TR>
```

```json
{
  "agent_id": "agt_business_001",
  "tool_id": "mcp:contract-server/get_contract",
  "arguments": {
    "contract_no": "HT-2026-001"
  }
}
```

### 3.4.2 `MCP 网关 -> 策略中心 : resolve-by-tools + query strategies + resolve-by-codes`

含义：
- MCP 网关先按当前待调用工具反查所需权限点。
- 再查询当前 Agent 针对这些权限点定义了哪些策略。
- 最后再按 `TR` 中权限点反查工具集合。

参考请求 1：按工具反查权限点

```json
{
  "requiredTools": [
    "mcp:contract-server/get_contract"
  ]
}
```

参考请求 2：按 Agent + 权限点查询策略

```json
{
  "agentId": "agt_business_001",
  "permissionPointCodes": [
    "erp:contract:r"
  ]
}
```

参考请求 3：按 `TR` 中权限点反查工具集合

```json
{
  "permissionPointCodes": [
    "erp:contract:r"
  ]
}
```

### 3.4.3 `MCP 网关 -> MCP 服务 : 通过 TR 校验和策略判断后路由调用`

含义：
- MCP 网关先检查：
  - 当前工具所需权限点是否已经在 `TR.agency_user.consented_scopes` 中
- 再检查：
  - 当前 `agent_id + permissionPointCode` 对应的 Agent 策略是否允许当前用户
- 然后再把 `TR` 中权限点反查成工具集合。
- 只有当当前请求工具出现在该工具集合中时，才真正路由到 MCP 服务。

### 3.4.4 `MCP 服务 -> MCP 网关 : 返回工具结果`

含义：
- 具体工具执行完成，返回业务数据。

参考响应：

```json
{
  "contractNo": "HT-2026-001",
  "summary": "该合同当前状态为审批中"
}
```

### 3.4.5 `MCP 网关 -> 业务 Agent : 返回数据`

含义：
- MCP 网关把工具结果回给业务 Agent。

### 3.4.6 `业务 Agent -> 用户 : 返回答案`

含义：
- 业务 Agent 再把原始结果组织成面向用户的回答。

## 3.5 本地无可用 TR 分支

### 3.5.1 `业务 Agent -> Agent 网关 : POST /gw/token/resource-token`

含义：
- 这是业务 Agent 正式向网关申请 `TR` 的入口。
- 也是 `gw_session_token` 第一次在授权阶段被明确使用的地方。

这一步表达的是：
- 我是 `agt_business_001`
- 我当前这个网站会话绑定的网关用户是谁，由 `gw_session_token` 表示
- 我现在需要调用这些工具
- 如果需要用户补充授权，授权完成后请回到这个 `return_url`

参考请求：

```http
POST /gw/token/resource-token
Authorization: Bearer gwst_abc_xyz
Content-Type: application/json
```

```json
{
  "agent_id": "agt_business_001",
  "required_tools": [
    "mcp:contract-server/get_contract"
  ],
  "return_url": "https://business-agent.huawei.com/agent.html",
  "state": "st_auth_001"
}
```

### 3.5.2 `Agent 网关 -> 策略中心 : resolve-by-tools(required_tools)`

含义：
- 网关自己不让业务 Agent 理解权限点。
- 所以网关把 `required_tools` 交给策略中心，翻译成权限点 code。
- 这里的工具与权限点映射不带 `agent` 维度，因此请求体不包含 `agentId`。

参考请求：

```json
{
  "requiredTools": [
    "mcp:contract-server/get_contract"
  ]
}
```

### 3.5.3 `策略中心 -> Agent 网关 : 返回 requiredPermissionPointCodes`

含义：
- 策略中心告诉网关：
  - 这些工具对应哪些权限点
  - 这些权限点的中文描述是什么

参考响应：

```json
{
  "requiredPermissionPointCodes": [
    "erp:contract:r"
  ],
  "permissionPoints": [
    {
      "code": "erp:contract:r",
      "displayNameZh": "ERP 合同的可读权限"
    }
  ]
}
```

### 3.5.4 `网关已有有效 TR -> 直接返回 TR`

含义：
- 如果网关侧已经存在覆盖当前权限点的有效 `TR`，就不需要再让浏览器走授权页。
- 这时网关可以直接把 `TR` 回给业务 Agent。

参考响应：

```json
{
  "status": "TOKEN_READY",
  "access_token": "<TR>",
  "expires_in": 3600
}
```

## 3.6 需要用户参与授权分支

### 3.6.1 `Agent 网关 -> 业务 Agent : 返回 redirect_url + request_id`

含义：
- 网关判断当前没有可直接使用的 `TR`，因此需要用户参与授权。
- 网关把“下一跳地址” `redirect_url` 和这次事务编号 `request_id` 回给业务 Agent。

参考响应：

```json
{
  "status": "REDIRECT_REQUIRED",
  "request_id": "req_001",
  "redirect_url": "https://agent-gateway.huawei.com/gw/auth/authorize?request_id=req_001"
}
```

说明：
- `redirect_url` 是“先去网关开始授权”
- `return_url` 是“授权完再回业务 Agent”

### 3.6.2 `用户 -> Agent 网关 : GET /gw/auth/authorize`

含义：
- 浏览器根据 `redirect_url` 进入网关授权入口。

参考请求：

```http
GET /gw/auth/authorize?request_id=req_001
Cookie: gw_session_id=gws_123
```

### 3.6.3 `Agent 网关 -> 用户 : 302 到 IDaaS /authorize?scope=<requiredPermissionPointCodes>`

含义：
- 网关根据 `request_id` 找到本次申请缺失的权限点。
- 然后把浏览器送去 IDaaS，让用户正式勾选并确认这些权限点。

参考响应：

```http
302 Location:
https://idaas.huawei.com/oauth2/authorize
  ?response_type=code
  &client_id=gw_client_001
  &redirect_uri=https%3A%2F%2Fagent-gateway.huawei.com%2Fgw%2Fauth%2Fconsent%2Fcallback
  &scope=erp%3Acontract%3Ar
  &state=gwst_auth_001
```

### 3.6.4 `用户 -> IDaaS : 完成授权`

含义：
- 用户在 IDaaS 页面看到“权限点 code + 中文描述”，然后勾选确认。

参考数据：

```text
授权权限点：
  erp:contract:r -> ERP 合同的可读权限
```

### 3.6.5 `IDaaS -> 用户 : 回调 /gw/auth/consent/callback`

含义：
- 用户确认授权后，IDaaS 仍然先回调到网关。

参考响应：

```http
302 Location:
https://agent-gateway.huawei.com/gw/auth/consent/callback
  ?code=code_consent_001
  &state=gwst_auth_001
```

### 3.6.6 `用户 -> Agent 网关 : 回网关`

含义：
- 浏览器把业务授权阶段的 `code + state` 带回网关。

### 3.6.7 `Agent 网关 -> IDaaS : code 换 Tc`

含义：
- 网关用这次授权的 code 换取 `Tc`。
- 此时 `Tc` 已经带上用户刚授权给当前 Agent 的权限点对象数组。

参考响应：

```json
{
  "access_token": "<Tc>",
  "consented_scopes": [
    {
      "code": "erp:contract:r",
      "displayNameZh": "ERP 合同的可读权限"
    }
  ]
}
```

### 3.6.8 `Agent 网关 -> IAM : 申请 T1`

含义：
- 网关以当前 Agent 身份申请机机令牌 `T1`。
- 请求头里的 `Authorization` 后续由 `IAM SDK` 获取，当前阶段可先走配置文件。
- `delegator_account_name`、`delegator_appid`、`agent_id` 后续由数据库提供，当前阶段可先走配置文件并保留 `TODO`。

参考响应：

```json
{
  "message": "OK",
  "code": "201",
  "enterprise": "11111111111111111111111111111111",
  "access_token": "<T1>",
  "expires_at": "2026-04-03T14:24:48+08:00",
  "expires_in": 86399,
  "token_id": "1234",
  "token_type": "IAM_AI_SERVICE",
  "expires_on": 1775197488000
}
```

### 3.6.9 `Agent 网关 -> IAM : 申请 TR`

含义：
- 网关用 `Tc + T1` 申请最终的资源令牌 `TR`。
- `TR` 会在 `agency_user.consented_scopes` 中包含用户授权给当前 Agent 的权限点对象集合。

参考响应：

```json
{
  "message": "OK",
  "code": "201",
  "enterprise": "11111111111111111111111111111111",
  "access_token": "<TR>",
  "expires_at": "2026-04-03T14:24:48+08:00",
  "expires_in": 86399,
  "token_id": "1234",
  "token_type": "IAM_AI_SERVICE",
  "refresh_token": "11111111113",
  "expires_on": 1775197488000
}
```

### 3.6.10 `Agent 网关 -> 用户 : 回业务 Agent`

含义：
- 网关完成 `TR` 编排后，再把浏览器送回业务 Agent 原页面。

参考响应：

```http
302 Location:
https://business-agent.huawei.com/agent.html?state=st_auth_001
```

### 3.6.11 `用户 -> 业务 Agent : 恢复原请求`

含义：
- 业务 Agent 前端从 `sessionStorage` 恢复刚才被打断的消息，再重新发送一次。

### 3.6.12 `业务 Agent -> Agent 网关 : 再次取 TR`

含义：
- 这是 `gw_session_token` 在授权阶段第二次被明确使用的地方。
- 业务 Agent 现在带着：
  - `gw_session_token`
  - `request_id`

告诉网关：
- 还是刚才那个已经登录的用户
- 继续拿回刚才授权事务对应的 `TR`

参考请求：

```http
POST /gw/token/resource-token
Authorization: Bearer gwst_abc_xyz
Content-Type: application/json
```

```json
{
  "request_id": "req_001"
}
```

### 3.6.13 `Agent 网关 -> 业务 Agent : 返回 TR`

含义：
- 网关返回刚才已经准备好的最终 `TR`。

参考响应：

```json
{
  "status": "TOKEN_READY",
  "access_token": "<TR>",
  "expires_in": 3600
}
```

### 3.6.14 后续再次进入 `MCP 网关` 调用链路

含义：
- 之后流程回到“本地已有可用 `TR` 分支”：
  - 业务 Agent 带 `TR` 调 MCP 网关
  - MCP 网关先按当前工具反查所需权限点
  - 再对这些权限点做 Agent 策略判断
  - 再按 `TR` 中权限点反查工具集合
  - 再校验当前请求工具是否在该工具集合内
  - MCP 服务返回结果
  - 业务 Agent 返回最终答案

## 4. 最值得单独强调的几个点

### 4.1 `gw_session_token` 在哪里被使用

最关键的两个使用点是：

1. `base 登录完成后`
   - 业务 Agent 用它建立本地 `site_session`
2. `业务授权阶段申请 / 再次申请 TR 时`
   - 业务 Agent 带它向网关证明：
     - 当前网站会话对应的是哪个网关已登录用户

### 4.2 `return_url` 和 `redirect_url` 的区别

- `redirect_url`
  - 网关返回给业务 Agent 的“下一跳地址”
  - 用于让浏览器先去网关开始授权
- `return_url`
  - 业务 Agent 传给网关的“最后回跳地址”
  - 用于告诉网关流程结束后浏览器要回到哪个业务页面

### 4.3 图里省略但容易被问到的回包

在时序图里通常会省略这三类详细回包：

- `IDaaS -> Agent 网关` 的 `Tc`
- `IAM -> Agent 网关` 的 `T1`
- `IAM -> Agent 网关` 的 `TR`

正式实现时，这三步都会有明确响应结构，只是在主流程图里不一定全部展开。

## 5. 一句话版讲解口径

如果只想用最短的话讲给别人听，可以直接说：

```text
用户先通过 Agent 网关完成 base 登录；
网关创建 gw_session，并把 gw_session_token 交给业务 Agent；
业务 Agent 用 gw_session_token 建立自己的 site_session；
后续业务 Agent 带 gw_session_token 和 required_tools 向网关申请 TR；
网关负责把工具翻译成权限点、必要时带用户补充授权，然后返回 TR；
业务 Agent 再带 TR 调 MCP 网关，MCP 网关先校验 TR 范围，再校验 Agent 策略。
```
