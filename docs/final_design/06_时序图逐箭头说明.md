# 时序图逐箭头说明

本文配套说明 [02_引入Agent网关版方案.md](./02_引入Agent网关版方案.md) 中的两张主流程图。当前正式方案采用模式 A：`IDaaS` 判断用户是否已登录，`Agent 网关` 不维护长期用户登录态，只维护短期事务和一次性结果凭据。

更细的拆分阅读：

- [flow_details/01_base_login_ticketST.md](./flow_details/01_base_login_ticketST.md)
- [flow_details/02_resource_token_ticket.md](./flow_details/02_resource_token_ticket.md)
- [flow_details/03_one_time_ticket_security.md](./flow_details/03_one_time_ticket_security.md)

## 1. 统一参考数据

```text
user_id = z01062668
username = 张三

agent_id = agt_business_001
app_id = com.huawei.business.agent
client_id = gw_client_001

site_session_id = site_789

ticketST = st_login_001
request_id = req_001
token_result_ticket = trt_001
state = st_auth_001

return_url = https://business-agent.huawei.com/agent

required_tools = [
  mcp:contract-server/get_contract
]

requiredPermissionPointCodes = [
  erp:contract:r
]
```

## 2. base 登录阶段逐箭头说明

### 2.1 `用户 -> 业务 Agent : GET /agent`

用户第一次打开业务 Agent 页面。业务 Agent 先检查自己域名下有没有 `site_session_id` Cookie。

```http
GET /agent
```

### 2.2 `业务 Agent -> 业务 Agent : 检查 site_session_id`

业务 Agent 只检查自己的本地站点会话，不检查 Agent 网关会话。命中 `site_session` 就直接返回页面；未命中才进入网关登录。

### 2.3 `业务 Agent -> 用户 : 302 到 /gw/auth/login`

业务 Agent 把浏览器送到 Agent 网关登录入口。

```http
302 Location:
https://agent-gateway.huawei.com/gw/auth/login
  ?agent_id=agt_business_001
  &return_url=https%3A%2F%2Fbusiness-agent.huawei.com%2Fagent
  &state=st_login_001
```

`state` 是业务 Agent 自己的页面恢复状态，用于回跳后恢复前端上下文。

### 2.4 `用户 -> Agent 网关 : GET /gw/auth/login`

浏览器正式进入 Agent 网关。网关校验 `agent_id` 和 `return_url`，创建 `pending_base_login`。

### 2.5 `Agent 网关 -> 用户 : 302 到 IDaaS /authorize`

网关把浏览器送到 IDaaS。

```http
302 Location:
https://idaas.huawei.com/oauth2/authorize
  ?response_type=code
  &client_id=gw_client_001
  &agent_id=agt_business_001
  &redirect_uri=https%3A%2F%2Fagent-gateway.huawei.com%2Fgw%2Fauth%2Fbase%2Fcallback
  &scope=base
  &state=gwst_login_001
```

这里的 `gwst_login_001` 是网关内部 `gw_state`，不是业务 Agent 的 `state`。

### 2.6 `用户 -> IDaaS : GET /oauth2/authorize`

浏览器收到 302 后，会真正访问 IDaaS 的 `/oauth2/authorize`。这是浏览器行为，不是 Agent 网关服务端代调 IDaaS。

### 2.7 `IDaaS -> 用户 : 展示登录页或快速回调`

如果浏览器已有 IDaaS 登录 Cookie，IDaaS 可以快速回跳；如果没有，则展示登录页让用户输入账号密码。

### 2.8 `用户 -> IDaaS : 提交登录信息（如需要）`

如果 IDaaS 展示了登录页，用户提交工号、密码或其他认证材料。如果已有登录态，这一步可以省略。

### 2.9 `IDaaS -> 用户 : 302 回 /gw/auth/base/callback`

IDaaS 登录完成后把浏览器回调到 Agent 网关。

```http
302 Location:
https://agent-gateway.huawei.com/gw/auth/base/callback
  ?code=code_base_001
  &state=gwst_login_001
```

### 2.10 `用户 -> Agent 网关 : GET /gw/auth/base/callback`

浏览器把 `code + gw_state` 带回 Agent 网关。网关用 `gw_state` 找回 `pending_base_login`。

### 2.11 `Agent 网关 -> Agent 网关 : 生成 ticketST`

网关此时不立即用 `code` 换用户信息，而是生成一次性 `ticketST`，并把 `code`、`client_id`、`redirect_uri`、`agent_id`、`return_url` 绑定到这张票据上。

参考数据：

```text
ticketST = st_login_001
ticketST -> code_base_001
```

### 2.12 `Agent 网关 -> 用户 : 回业务 Agent，只带 ticketST`

网关通过浏览器回跳业务 Agent。

```http
302 Location:
https://business-agent.huawei.com/agent
  ?ticketST=st_login_001
  &state=st_login_001
```

这一跳不携带 `Tc`、`TR`、用户工号、用户名或用户 UUID。

### 2.13 `用户 -> 业务 Agent : 回原页面`

业务 Agent 收到 `ticketST`，但前端不能直接把它当用户信息使用。

```http
GET /agent?ticketST=st_login_001&state=st_login_001
```

### 2.14 `业务 Agent -> Agent 网关 : POST /gw/auth/ticket/exchange`

业务 Agent 后端用 `ticketST + agent_id` 换取用户信息。

```json
{
  "agent_id": "agt_business_001",
  "ticketST": "st_login_001"
}
```

### 2.15 `Agent 网关 -> IDaaS : code 换 Tc 短 access_token`

网关在交换接口内取出 `ticketST` 绑定的 `code`，结合 Agent 网关的 `client_id + client_secret` 调用 IDaaS `/oauth2/token`。`redirect_uri` 是否传递取决于 IDaaS 是否要求校验 callback 地址。这一步返回的是 `Tc` 对应的短令牌 `access_token`。

### 2.16 `Agent 网关 -> IDaaS : 用 Tc 调用户信息接口`

网关拿 `Tc access_token` 调 IDaaS 用户信息接口，获取可信用户信息。获取用户信息只需要传 `access_token`。

### 2.17 `IDaaS -> Agent 网关 : 返回用户信息`

IDaaS 返回用户工号、用户名、UUID 等基础信息。

### 2.18 `Agent 网关 -> 业务 Agent : 返回用户信息`

Agent 网关只把用户信息返回给业务 Agent 后端，不把 `Tc access_token` 返回给业务 Agent。

### 2.19 `业务 Agent -> 业务 Agent : 建立 site_session`

业务 Agent 后端拿到用户信息后创建自己的 `site_session`。

```text
site_session_id = site_789
site_session = {
  user_id: z01062668,
  username: 张三
}
```

```http
Set-Cookie: site_session_id=site_789; Path=/; HttpOnly; SameSite=Lax
```

## 3. 业务授权 + 获取 TR 阶段逐箭头说明

### 3.1 `用户 -> 业务 Agent : POST /chat/send`

用户发起真实业务请求，例如“帮我看一下这个合同”。

### 3.2 `业务 Agent -> 业务 Agent : 判断 required_tools`

业务 Agent 判断本次需要调用哪些 MCP 工具。

```text
required_tools = [
  mcp:contract-server/get_contract
]
```

### 3.3 `业务 Agent -> 业务 Agent : 检查本地 TR 缓存`

业务 Agent 先检查自己的 `tr_cache`。如果有未过期且覆盖当前工具的 `TR`，直接调用 MCP 网关；如果没有，才向 Agent 网关申请授权。

### 3.4 `业务 Agent -> Agent 网关 : POST /gw/token/resource-token`

业务 Agent 后端提交本次需要的工具集合。

```json
{
  "agent_id": "agt_business_001",
  "required_tools": [
    "mcp:contract-server/get_contract"
  ],
  "return_url": "https://business-agent.huawei.com/agent",
  "state": "st_auth_001",
  "subject_hint": {
    "userId": "z01062668"
  }
}
```

`subject_hint` 只作为 IDaaS 登录提示，不是可信身份。

### 3.5 `Agent 网关 -> 策略中心 : resolve-by-tools`

网关把 `required_tools` 解析成权限点 code。

```json
{
  "requiredTools": [
    "mcp:contract-server/get_contract"
  ]
}
```

### 3.6 `Agent 网关 -> 业务 Agent : 返回 redirect_url + request_id`

网关创建 `pending_auth_transaction`，返回浏览器授权入口。

```json
{
  "status": "REDIRECT_REQUIRED",
  "request_id": "req_001",
  "redirect_url": "https://agent-gateway.huawei.com/gw/auth/authorize?request_id=req_001"
}
```

### 3.7 `业务 Agent -> 用户 : 跳转 redirect_url`

业务 Agent 前端把原消息写入 `sessionStorage`，再跳转到 `redirect_url`。

### 3.8 `用户 -> Agent 网关 : GET /gw/auth/authorize`

浏览器进入网关授权入口。网关通过 `request_id` 找到本次事务。

### 3.9 `Agent 网关 -> 用户 : 302 到 IDaaS /authorize`

网关把浏览器送到 IDaaS 授权页。

```http
302 Location:
https://idaas.huawei.com/oauth2/authorize
  ?response_type=code
  &client_id=gw_client_001
  &agent_id=agt_business_001
  &redirect_uri=https%3A%2F%2Fagent-gateway.huawei.com%2Fgw%2Fauth%2Fconsent%2Fcallback
  &scope=erp%3Acontract%3Ar
  &state=gwst_auth_001
  &login_hint=z01062668
```

### 3.10 `用户 -> IDaaS : GET /oauth2/authorize`

浏览器收到 302 后，会真正访问 IDaaS 授权地址。首次非 base 授权不会只是“闪一下”，通常会进入授权页。

### 3.11 `IDaaS -> 用户 : 展示授权页`

IDaaS 展示本次申请的权限点，例如 `erp:contract:r / ERP 合同的可读权限`，让用户理解 Agent 申请访问什么能力。

### 3.12 `用户 -> IDaaS : 勾选并确认授权`

用户确认授权后，IDaaS 才会生成授权码。只有在用户之前已经对同一 Agent、同一权限点做过长期授权，且 IDaaS 策略允许静默通过时，这里才可能快速回跳。

### 3.13 `IDaaS -> 用户 : 302 回 /gw/auth/consent/callback`

IDaaS 完成授权后，不是服务端直接调 Agent 网关，而是让浏览器 302 回 Agent 网关 callback。

### 3.14 `用户 -> Agent 网关 : GET /gw/auth/consent/callback`

浏览器把 `code + gw_state` 带回 Agent 网关。

### 3.15 `Agent 网关 -> IDaaS : code 换 Tc`

网关拿到 `Tc`。`Tc.consented_scopes` 表达用户本次授权给 Agent 的权限点对象数组。

### 3.16 `Agent 网关 -> IAM : 申请 T1`

网关以 Agent 身份向 IAM 申请 `T1`。

### 3.17 `Agent 网关 -> IAM : 申请 TR`

网关用 `Tc + T1` 向 IAM 申请 `TR`。

### 3.18 `Agent 网关 -> 用户 : 回业务 Agent，只带 token_result_ticket`

网关生成一次性 `token_result_ticket`，并通过浏览器回跳业务 Agent。

```http
302 Location:
https://business-agent.huawei.com/agent
  ?token_result_ticket=trt_001
  &request_id=req_001
  &state=st_auth_001
```

这一跳不携带 `TR`。

### 3.19 `业务 Agent -> Agent 网关 : POST /gw/token/result/exchange`

业务 Agent 后端用 `token_result_ticket` 换取 `TR`。

```json
{
  "agent_id": "agt_business_001",
  "request_id": "req_001",
  "token_result_ticket": "trt_001"
}
```

### 3.20 `业务 Agent -> 业务 Agent : 校验用户一致并缓存 TR`

业务 Agent 校验 `TR.agency_user` 与当前 `site_session` 用户一致。校验通过后写入 `tr_cache`。

### 3.21 `业务 Agent -> MCP 网关 : 带 TR 调 MCP`

业务 Agent 带 `TR` 调用 MCP 网关。

### 3.22 `MCP 网关 -> 策略中心 : resolve-by-tools(currentTool)`

MCP 网关按当前待调用工具反查所需权限点。

### 3.23 `MCP 网关 -> 策略中心 : agent-strategies/query`

MCP 网关查询当前 Agent 对这些权限点的策略，并在本地执行策略判断。

### 3.24 `MCP 网关 -> 策略中心 : resolve-by-codes(TR scopes)`

MCP 网关按 `TR.agency_user.consented_scopes[].code` 反查工具集合。

### 3.25 `MCP 网关 -> MCP 服务 : 路由调用`

只有同时满足“当前工具所需权限点在 TR 中、策略放行、当前工具属于 TR 反查出的工具集合”，MCP 网关才调用后端 MCP 服务。

## 4. 规则总结

- Agent 网关不维护长期用户登录态。
- 用户是否已登录由 IDaaS 判断。
- 业务 Agent 用 `site_session` 维护自己的站点登录态。
- base 登录结果通过 `ticketST` 后端交换。
- 业务授权结果通过 `token_result_ticket` 后端交换。
- 浏览器 URL 不出现 `Tc`、`TR`、用户信息。
- `TR` 复用由业务 Agent 本地 `tr_cache` 决定。
- MCP 网关运行时仍必须做 `TR` 范围校验和 Agent 策略判断。
