## 2. 一句话主线

- 当前正式方案是：业务 Agent 只理解 `required_tools`、`TR` 和 `redirect_url`；`Agent 网关` 统一负责登录、授权、`Tc / T1 / TR` 编排；`策略中心` 维护权限点定义、权限点与工具绑定关系、Agent 策略；`MCP 网关` 先校验 `TR` 授权范围，再执行 Agent 策略判断，最终决定当前工具是否允许调用。

## 3. 当前推荐架构

- 用户浏览器 ↔ 业务 Agent
- 用户浏览器 ↔ Agent 网关
- 业务 Agent ↔ Agent 网关
- Agent 网关 ↔ IDaaS
- Agent 网关 ↔ IAM
- Agent 网关 ↔ 策略中心
- 业务 Agent ↔ MCP 网关
- MCP 网关 ↔ MCP 服务

## 4. 各模块职责速记

### 4.1 业务 Agent

- 判断本次请求需要哪些 `required_tools`
- 本地维护 `site_session`
- 本地维护当前会话下的 `current_tr + covered_tools + coveredPermissionPointCodes`
- 调用 `POST /gw/token/resource-token`
- 带 `TR` 调用 `MCP 网关`

### 4.2 Agent 网关

- 统一承接 OAuth2 跳转和 callback
- 调 `IDaaS /oauth2/token` 获取基础登录结果或 `Tc`
- 调 `IAM` 获取 `T1`
- 用 `Tc + T1` 获取 `TR`
- 把 `required_tools` 翻译成 `requiredPermissionPointCodes`
- 维护 `gw_session / gw_auth_context / pending_base_login / pending_auth_transaction`

### 4.3 策略中心

- 维护权限点定义
- 维护权限点与工具绑定关系
- 维护 Agent 策略
- 向 Agent 网关提供 `resolve-by-tools`
- 向 MCP 网关提供 `resolve-by-codes` 和 `agent-strategies/query`

### 4.4 MCP 网关 / MCP 服务

- `MCP 网关` 接受 `TR`
- 先校验当前请求工具所需权限点是否在 `TR.authorizedPermissionPoints` 内
- 再根据 Agent 策略判断当前用户是否允许使用这些权限点
- 路由到具体 `MCP 服务`

## 5. 核心对象速记

- `required_tools`：业务 Agent 侧上传的本次所需工具集合
- `permissionPoint`：权限点对象，至少包含 `code + displayNameZh`
- `Tc`：用户令牌，携带 `authorizedPermissionPoints`
- `T1`：Agent 令牌
- `TR`：资源令牌，携带 `authorizedPermissionPoints`
- `gw_session_token`：网关回给业务 Agent 的不透明会话引用
- `site_session_id`：业务 Agent 自己的网站登录态
- `request_id`：业务 Agent 与网关之间“获取 TR”流程的外部流程号
- `gw_state`：网关内部 OAuth/browser callback 事务号，不对业务 Agent 暴露

## 6. 两段主流程速记

### 6.1 首次打开页面

- 业务 Agent 检查本地 `site_session`
- 无登录态则跳转到 Agent 网关
- 网关完成 `base` 登录
- 网关回业务 Agent，并带 `gw_session_token`
- 业务 Agent 创建本地 `site_session`

### 6.2 首次访问业务资源

- 业务 Agent 判断 `required_tools`
- 若本地无可用 `TR`，调用 `POST /gw/token/resource-token`
- Agent 网关通过策略中心把 `required_tools` 解析成 `requiredPermissionPointCodes`
- 若需要用户授权，网关返回 `redirect_url + request_id`
- 用户完成授权后，业务 Agent 带 `request_id` 再次请求网关拿 `TR`
- 业务 Agent 带 `TR` 调 `MCP 网关`
- `MCP 网关` 先做 `TR` 范围校验，再做 Agent 策略判断

## 7. 当前最小接口心智

- 业务 Agent 真正长期对接的核心接口：
  - `POST /gw/token/resource-token`
- 浏览器侧统一入口：
  - `/gw/auth/login`
  - `/gw/auth/authorize`
- 网关内部依赖：
  - `POST /oauth2/token`
  - `POST /iam/projects/{proxy_project_id}/assume_agent_token`
  - `POST /iam/auth/resource-token`
  - `POST /internal/v1/permission-points/resolve-by-tools`
  - `POST /internal/v1/permission-points/resolve-by-codes`
  - `POST /internal/v1/agent-strategies/query`

## 8. 当前确定的关键设计决策

- `01` 是参考方案，`02/03/04/05` 是当前正式方案
- 业务 Agent 不再理解 `scope/code`
- 业务 Agent 不直接对接 `IDaaS / IAM`
- `TR` 是用户授权给 Agent 的上限边界
- 策略是 Agent 所有者对不同用户功能开放范围的二次控制
- 权限点定义由 `MCP 网关` 上报策略中心
- Agent 策略由 `Agent 网关` 上报策略中心
- `request_id` 对业务 Agent 外显，`gw_state` 只保留在网关内部
- `TR` 本地缓存统一采用覆盖式缓存，不再保留按单次工具集合建正式缓存模型
