## 2. 一句话主线

当前正式方案采用模式 A：`IDaaS` 负责判断用户是否已登录和是否需要展示授权页，`Agent 网关` 只负责浏览器跳转编排、`code -> Tc -> T1 -> TR` 编排、权限点解析，以及短期一次性凭据交付；`业务 Agent` 维护自己的 `site_session` 和 `TR` 缓存；`MCP 网关` 在运行时做 `TR` 范围校验和 Agent 策略判断。

运行前置条件：注册 Agent 后，外部系统已经调用 IDaaS 委托授权接口，建立 `Agent 网关 client_id` 与 `业务 Agent agent_id` 的信任关系；这里的 `agent_id` 在全系统共用同一个值，也就是 Agent 网关注册表中的 `agent_id`。

## 3. 当前推荐架构

- 用户浏览器 ↔ 业务 Agent
- 用户浏览器 ↔ Agent 网关
- 用户浏览器 ↔ IDaaS
- 业务 Agent ↔ Agent 网关
- Agent 网关 ↔ IDaaS
- Agent 网关 ↔ IAM
- Agent 网关 ↔ 策略中心
- 业务 Agent ↔ MCP 网关
- MCP 网关 ↔ 策略中心
- MCP 网关 ↔ MCP 服务

## 4. 各模块职责速记

### 4.1 业务 Agent

- 提供页面和业务接口。
- 本地维护 `site_session`。
- 本地维护 `tr_cache`，缓存当前会话可复用的 `TR`。
- 判断本次请求需要哪些 `required_tools`。
- 使用 `ticketST` 向 Agent 网关换取登录用户信息。
- 使用 `token_result_ticket` 向 Agent 网关换取 `TR`。
- 带 `TR` 调用 `MCP 网关`。

### 4.2 Agent 网关

- 统一承接 OAuth2 跳转和 callback。
- 调 `IDaaS /oauth2/token` 用 `code` 换 `Tc` 短令牌。
- 调 IDaaS 用户信息接口获取基础用户信息。
- 调 `IAM` 获取 `T1`。
- 用 `Tc + T1` 获取 `TR`。
- 把 `required_tools` 翻译成 `requiredPermissionPointCodes`。
- 维护短期事务：`pending_base_login`、`pending_auth_transaction`。
- 维护一次性结果凭据：`ticketST`、`token_result_ticket`。
- 不维护长期用户登录态，不把 `Tc / TR / 用户信息` 放到浏览器 URL。

### 4.3 策略中心

- 维护权限点定义。
- 维护权限点与工具绑定关系。
- 维护 Agent 与权限点订阅映射。
- 维护 Agent 策略。
- 通过 `batch-upsert + status` 完成权限点和策略的停用/恢复。
- 权限点由 `permissionPointCode` 全系统唯一标识，上报时必须携带 `enterprise` 和 `appId`。
- 权限点查询必须携带 `enterprise`，`appId` 可选；不传 `appId` 表示查询该企业下全部应用的权限点。
- 向 Agent 网关提供 `resolve-by-tools`。
- 向 MCP 网关提供 `resolve-by-tools`、`resolve-by-codes` 和 `agent-strategies/query`。
- 工具与权限点映射不带 `agent` 维度。

### 4.4 MCP 网关 / MCP 服务

- `MCP 网关` 接受 `TR`。
- 先按当前待调用工具反查所需权限点。
- 再校验这些权限点是否在 `TR.agency_user.consented_scopes` 中。
- 再结合 Agent 策略判断当前用户是否允许使用这些权限点。
- 再根据 `TR` 中权限点反查可访问工具集合。
- 最后校验当前请求工具是否在该工具集合内，并路由到具体 `MCP 服务`。

## 5. 核心对象速记

- `required_tools`：业务 Agent 上传的本次所需工具集合。
- `permissionPoint`：权限点对象，至少包含 `code + displayNameZh`。
- `Tc`：用户令牌，携带 `consented_scopes`。
- `T1`：Agent 令牌。
- `TR`：资源令牌，在 `agency_user.consented_scopes` 中携带权限点对象数组。
- `site_session_id`：业务 Agent 自己的网站登录态 Cookie。
- `ticketST`：base 登录完成后，浏览器回跳业务 Agent 时携带的一次性票据。
- `token_result_ticket`：授权完成后，浏览器回跳业务 Agent 时携带的一次性取件凭据。
- `request_id`：业务 Agent 与网关之间“获取 TR”流程的外部流程号。
- `gw_state`：网关内部 OAuth/browser callback 事务号，不对业务 Agent 暴露。

## 6. 两段主流程速记

### 6.1 首次打开页面

- 业务 Agent 检查本地 `site_session`。
- 无登录态则跳转到 Agent 网关 `/gw/auth/login`。
- Agent 网关跳转到 IDaaS。
- IDaaS 判断是否需要用户输入账号密码。
- IDaaS 回调 Agent 网关。
- Agent 网关暂存本次 base 登录 `code`，生成 `ticketST`。
- 浏览器回到业务 Agent，只携带 `ticketST + state`。
- 业务 Agent 后端调用 `/gw/auth/ticket/exchange`。
- Agent 网关在交换接口内用 `code` 换 `Tc` 短令牌，再用 `Tc` 调 IDaaS 用户信息接口。
- 业务 Agent 创建本地 `site_session`。

### 6.2 首次访问业务资源

- 业务 Agent 判断 `required_tools`。
- 若本地无可用 `TR`，调用 `POST /gw/token/resource-token`。
- Agent 网关通过策略中心把 `required_tools` 解析成 `requiredPermissionPointCodes`。
- Agent 网关创建 `pending_auth_transaction`，返回 `redirect_url + request_id`。
- 用户浏览器访问 `redirect_url` 并进入 IDaaS 授权。
- IDaaS 可根据登录 Cookie 或长期授权记录快速回调，也可展示授权确认页。
- Agent 网关完成 `code -> Tc -> T1 -> TR`，生成 `token_result_ticket`。
- 浏览器回到业务 Agent，只携带 `token_result_ticket + request_id + state`。
- 业务 Agent 后端调用 `/gw/token/result/exchange` 换取 `TR`。
- 业务 Agent 缓存 `TR` 并带 `TR` 调 `MCP 网关`。

## 7. 当前最小接口心智

- 浏览器侧入口：
  - `GET /gw/auth/login`
  - `GET /gw/auth/authorize`
- 业务 Agent 后端对接 Agent 网关：
  - `POST /gw/auth/ticket/exchange`
  - `POST /gw/token/resource-token`
  - `POST /gw/token/result/exchange`
- 网关内部依赖：
  - `POST /oauth2/token`
  - `POST /iam/projects/{proxy_project_id}/assume_agent_token`
  - `POST /iam/auth/resource-token`
  - `POST /internal/v1/permission-points/query`
  - `POST /internal/v1/permission-points/resolve-by-tools`
  - `PUT /internal/v1/agent-permission-points/batch-upsert`
- MCP 网关运行时依赖：
  - `POST /internal/v1/permission-points/resolve-by-tools`
  - `POST /internal/v1/permission-points/resolve-by-codes`
  - `POST /internal/v1/agent-strategies/query`

## 8. 当前确定的关键设计决策

- `01` 是参考方案，`02/03/04/05` 是当前正式方案。
- 业务 Agent 不理解权限点内部结构，只上传 `required_tools`。
- 业务 Agent 不直接对接 `IDaaS / IAM`。
- `TR` 是用户授权给 Agent 的上限边界。
- 策略是 Agent 所有者对不同用户功能开放范围的二次控制。
- 权限点定义由 `MCP 网关` 上报策略中心。
- Agent 策略由 `Agent 网关` 上报策略中心。
- Agent 与权限点订阅映射由 `Agent 网关` 上报策略中心，当前只做注册面和管理面存储。
- 权限点 `permissionPointCode` 全系统唯一。
- 权限点上报时必须携带 `enterprise` 和 `appId`。
- 权限点查询必须携带 `enterprise`，`appId` 可选。
- 工具与权限点映射不带 `agent` 维度。
- 权限点和策略都采用状态删除，不新增独立 delete 接口。
- 只有 `ACTIVE` 权限点和 `ACTIVE` 策略参与运行时查询。
- Agent 网关不维护长期用户登录态，用户是否已登录由 IDaaS 判断。
- 运行时调用 IDaaS `/oauth2/authorize` 时，携带：
  - `client_id`：Agent 网关在 IDaaS 的 OAuth 客户端 ID
  - `agent_id`：当前业务 Agent 标识，全系统共用同一个值
- 浏览器 URL 中不出现 `Tc`、`TR`、用户信息，只出现一次性票据和业务 `state`。
- `ticketST` 和 `token_result_ticket` 都必须短 TTL、一次性使用、绑定 `agent_id` 和回跳白名单。
- `TR` 本地缓存统一放在业务 Agent，不再由 Agent 网关判断是否已有覆盖当前权限点的有效 `TR`。
